## 📊 Part 10: 指标定义字典（可验证标准）

### 核心指标定义（必须明确）

#### 1. AI匹配准确率（三段漏斗拆解）

**为什么要拆解**：
- 单一"准确率"口径容易被质疑
- 拆解成漏斗，更可证伪、更可优化

**三段漏斗定义**：

**阶段1：规则命中率**
```
定义：AI推荐的Top-3产品中，至少有1个产品满足银行准入规则

计算：
命中率 = 满足准入规则的客户数 / 总推荐客户数 × 100%

示例：
- 推荐100个客户
- 其中80个满足某个产品的准入规则（如：年开票≥100万）
- 命中率 = 80%

目标：
- V0：≥70%（与专家经验对齐）
- V1：≥80%（超越普通中介）
- V2：≥90%（接近顶级专家）
```

**阶段2：预授信通过率**
```
定义：银行给出预授信额度的比例（不含材料不全导致的拒绝）

计算：
预授信通过率 = 获得预授信的客户数 / 实际提交申请的客户数 × 100%

示例：
- 80个满足准入规则的客户提交申请
- 其中50个获得银行预授信
- 预授信通过率 = 50 / 80 = 62.5%

目标：
- V0：≥50%（验证AI推荐有效）
- V1：≥60%（优于行业平均）
- V2：≥70%（接近专家水平）
```

**阶段3：最终放款成功率**
```
定义：实际放款的比例（剔除客户主动放弃的）

计算：
放款成功率 = 实际放款客户数 / 获得预授信客户数 × 100%

示例：
- 50个获得预授信的客户
- 其中40个最终放款（10个主动放弃或材料不全）
- 放款成功率 = 40 / 50 = 80%

目标：
- V0：≥70%（大部分能走到最后）
- V1：≥75%（优化流程，减少放弃）
- V2：≥80%（接近行业最佳）
```

**综合准确率**（最终KPI）：
```
综合准确率 = 命中率 × 预授信通过率 × 放款成功率

示例：
- 命中率 80% × 预授信率 62.5% × 放款率 80% 
- = 40%（最终有40%的推荐客户成功放款）

目标：
- V0：≥30%（验证可行性）
- V1：≥40%（超越普通中介）
- V2：≥50%（接近顶级专家，行业顶尖水平约60%）
```

**为什么要三段拆解**：
1. **可定位问题**：知道卡在哪个环节
   - 命中率低 → Skill规则有问题
   - 预授信率低 → 银行政策理解不准
   - 放款率低 → 流程体验有问题
   
2. **可对比优化**：
   - A/B测试不同Skill版本
   - 对比不同银行产品的通过率
   
3. **可向投资人解释**：
   - 不是模糊的"80%准确率"
   - 而是"命中率90% × 预授信率70% × 放款率80% = 50%综合准确率"
```

---

#### 2. 企业自助成功率

**定义**：
```
企业在无人工干预的情况下，完成从"13字段输入"到"获得银行预授信"的完整流程比例
```

**分段统计**（更精确）：
```
- 流程完成率：完成13字段填写的比例
- 材料齐全率：上传所需材料的比例
- 提交成功率：成功提交到银行的比例
- 预审通过率：获得银行预授信的比例

自助成功率 = 预审通过率（最严格定义）
```

**Kill阈值说明**：

### DCC2026的合规架构（最小闭环）

**设计原则**：
```
1. 同意优先：收集前必须同意，用途明确
2. 最小必要：只收集必需的字段
3. 安全存储：加密+权限+审计
4. 可删可查：用户有权查询、修正、删除
5. 留痕可审：所有操作有日志
```

**具体实现**：

**Step 1：收集阶段**
```
用户导入客户数据前：
1. 弹窗《个人信息处理规则》
2. 明确告知：收集哪些字段、用于什么、保存多久
3. 单独勾选框："我已获得客户授权，同意DCC2026处理这些信息"
4. 记录同意时间+IP

代码示例（前端）：
<Checkbox>
  我确认已获得客户授权，同意DCC2026按照
  <a href="/privacy">《个人信息处理规则》</a>
  处理客户信息，用于贷款匹配和方案推荐
</Checkbox>
```

**Step 2：存储阶段**
```
敏感字段加密：
- 身份证号：AES-256加密
- 手机号：中间4位打码存储
- 银行账号：不存储（或加密存储）

权限控制：
- partner_id隔离（中介A看不到中介B的数据）
- 字段权限（普通用户看不到敏感字段）
- 操作日志（谁查了谁的数据）

示例（伪代码）：
def encrypt_sensitive_field(value, partner_id):
    key = get_encryption_key(partner_id)
    return AES.encrypt(value, key)
```

**Step 3：使用阶段**
```
最小化原则：
- AI匹配只用必需字段（13字段）
- 生成报告时脱敏（"客户A001"而非真实姓名）
- 导出数据时二次确认

审计日志：
{
  "timestamp": "2026-02-13T10:30:00Z",
  "action": "customer_data_export",
  "user": "admin@partner_a.com",
  "partner_id": "PARTNER_A",
  "customer_count": 50,
  "fields": ["name", "industry", "revenue"],
  "reason": "制作营销方案"
}
```

**Step 4：删除阶段**
```
用户权利：
- 客户要求删除 → 30天内完全删除
- 中介停止合作 → 60天内删除全部数据
- 数据过期 → 自动删除（2年无使用）

删除验证：
- 数据库删除
- 备份删除
- 日志脱敏（保留操作记录，删除个人信息）
```

### 数据资产沉淀策略（合规版）

- IF 行业=软件 AND 年收入>100万 AND 纳税>1万
  THEN 推荐"招商税贷"，额度估算=纳税×30

数据资产沉淀：
- 中介A贡献100个案例 → 提取20条规则
- 中介B贡献80个案例 → 提取15条规则
- 合并优化 → 35条规则（Skill系统）
- 所有中介受益，但看不到彼此的原始数据
```

**沉淀的是什么**：
- ✅ 产品Skill（YAML配置文件）
- ✅ 改造方案模板
- ✅ 匹配规则（if-then）
- ✅ 统计规律（不含个人信息）
