## 一句话

我们培养的不是"会写代码的人"，而是"用配置+AI构建产品，并能验收交付的人"——这套能力，换任何行业都通用，也是未来 3 年最能和资深程序员拉开差距的方向。

---

## 这份实习的本质是什么

传统实习：你写代码，项目上线，简历加一行。

这份实习：你参与建立一套**把业务规则变成可执行资产的生产线**，代码只是生产线的一部分。

具体来说，你要做的事是：

把银行贷款产品的规则（"年开票≥300万、征信查询≤6次、月均流水≥50万"），翻译成机器能读、能跑、能更新的配置文件（Skill），再写一段通用程序让所有配置自动跑通，最终输出一份"中介能看懂、企业能用上"的诊断报告。

这不是写 CRUD，是**设计规则资产**。

---

## 为什么这件事值得做

### 1. 规则资产是复利，代码不是

代码会被重构、迁移、替换。但你写的 Skill——准入条件、额度公式、解释模板——会越积越厚。每新增一个银行产品，只是加一个文件；每优化一条规则，系统立刻更聪明。

8 周后，你会留下 4 个可带走的硬资产：

| 资产 | 内容 | 价值 |
|---|---|---|
| 产品库 | products.json + 来源字段 + 版本号 | 可持续扩展的知识库 |
| 规则库 | skills/ 下若干产品 Skill（准入/计算/解释模板）| 可版本化、可审计的业务规则 |
| 回放用例 | 至少 10 个固定 case（输入→输出可对比）| 团队协作的共同语言 |
| 可运行基座 | API 可跑 + Docker 可起 + 一键回放验证 | 随时可演示的作品 |

这 4 样东西放到简历上，远比"参与开发了一个 SaaS 系统"有说服力。

### 2. 你学到的是方法论，不是贷款知识

这套能力的核心链路是：

**把需求拆成结构化（字段/规则/输出）→ 让 AI 产出（代码/配置/文档）→ 把产出变成可运行、可维护、可验收的交付物（测试/回放/质量闸门）**

无论换到医疗、法律、教育、供应链——任何需要把专家经验编码成产品的行业，这条链路都适用。

贷款是我们的实验田，方法论才是你拿走的东西。

### 3. 弯道超车的窗口期是真实存在的

现在绝大多数程序员还在用旧方式工作：理解需求 → 写代码 → 测试 → 交付。

你在这里训练的是新方式：理解规则 → 设计 Skill → 让 AI 实现 → 回放验收 → 可迭代交付。

代码量更少，但对业务理解和验收能力的要求更高。这个能力差，正是你能和有 10 年经验的程序员竞争的地方——他们写代码的经验你追不上，但"用配置+AI 构建产品"这件事，大家都在同一起跑线。

## 我们要训练的核心闭环

```
需求/规则
    ↓
Skill 配置（YAML，把产品规则写成机器能读的格式）
    ↓
AI 生成实现（Cursor/Claude Code 辅助写代码）
    ↓
回放验证（固定 case 输入 → 输出对比，确保结果对）
    ↓
可交付上线（pytest 通过 + ruff 通过 + Docker 可起）
```

这个闭环打通，你就具备了"AI 指挥官"的基础能力——不是"会问 AI"，而是"能让 AI 产出变成可用交付物"。

---

## 你会遇到的真实挑战

不说好听的，这 8 周有几个地方会让你难受：

**挑战 1：60% 的时间不在写代码**

你要花大量时间理解银行产品规则、设计 Skill 结构、验证输出是否符合业务逻辑。如果你只想写代码，会感觉"浪费时间"。但这正是培养"产品工程师"而非"纯程序员"的地方。

**挑战 2：单位陷阱会在某个时刻坑你**

系统里存在"万元 vs 元"的口径冲突（SSOT 定义万元，代码示例部分用元）。这会导致 1 万倍的计算错误。你必须在每个金额字段上都标注单位，并写专门的测试用例覆盖这个场景。不是坑，是训练。

**挑战 3：AI 生成的代码你必须看懂**

我们用 Cursor/Claude Code 辅助开发，但每一行代码你都要能解释。不通过质量闸门（pytest/ruff/bandit）的代码不算交付。"AI 写的"不是借口。

## 对应招聘文案里没讲清楚的几件事

**关于 Skill 是什么**

Skill 是一个 YAML 配置文件，把银行产品的准入条件、额度计算公式、报告解释模板写成结构化格式。改产品规则时，只改 Skill 文件，不动代码。这是 DCC2026 的核心架构，也是你主线工作的载体。

**关于回放用例**

回放用例是一组固定的"输入→预期输出"测试对。谁改了 Skill，跑一遍回放就知道有没有影响其他产品。这是团队协作的共同语言，不是可选的。

**关于"AI 指挥官"**

这不是口号。我们要的能力差异是：
- 普通实习生：会用 AI 写代码，但不会验证
- 我们要的：会用 AI 写代码，**也会让 AI 产出通过质量闸门变成可交付物**

面试和任务设计都围绕这个区别来考察。

**关于战略背景**

DCC2026 的底层逻辑是"把贷款专家的经验 AI 化，让普通企业主也能像专家一样搞定贷款"（详见 `DCC2026分析v2.3.md`）。你做的 Skill 系统，是实现这个目标的核心基础设施。理解这一层，你设计 Skill 的时候会有更清晰的取舍标准。

## 8 周结束，你能说什么

如果培训成功，你在面试时可以这样介绍自己的项目经历：

"我参与搭建了一个 AI 贷款诊断系统的核心基础设施。我的主要工作是把 20 多个真实银行产品的准入规则翻译成结构化的 Skill 配置文件，写代码让这些配置自动跑通，并用回放用例保证每次规则变更不影响已有逻辑。最终交付了产品库、规则库、回放用例套件和可一键启动的运行环境。这套方法论——用配置定义规则、用 AI 加速实现、用回放保证可验收——可以迁移到任何需要把专家知识编码成产品的场景。"

这段话，传统 8 周实习生说不出来。
