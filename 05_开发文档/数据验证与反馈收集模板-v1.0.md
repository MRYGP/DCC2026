## 1. 核心验证理念：用数据说话

### 1.1 验证什么？
```
不要问："你觉得这个系统好用吗？"
要问："你这周用了系统几次？"

不要问："这个功能有用吗？"
要问："你用这个功能做了什么具体的事情？"

不要问："你会推荐给朋友吗？"
要问："你最近有没有推荐给哪个朋友？为什么？"
```

### 1.2 验证指标

| 验证维度 | 关键问题 | 数据指标 | 目标值 |
|----------|----------|----------|--------|
| **是否在用** | 朋友真的在用吗？ | 活跃天数、诊断次数 | >3天/周，>5次/周 |
| **是否常用** | 是"日常工具"还是"一次性玩具"？ | 周使用频率、功能使用率 | 功能使用率>50% |
| **是否有用** | 真的帮到朋友了吗？ | 方案生成次数、成交反馈 | >2次/周，≥1单/朋友 |
| **是否推荐** | 朋友愿意推荐吗？ | 推荐意愿、实际推荐数 | >70%意愿，≥1个实际推荐 |

### 2.2 核心数据表设计

```sql
-- 使用记录表
CREATE TABLE usage_logs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    tenant_id INTEGER NOT NULL,
    user_id INTEGER,
    event_type TEXT NOT NULL,           -- page_view, feature_usage, diagnosis, plan_generation
    event_data TEXT,                    -- JSON格式的事件数据
    timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (tenant_id) REFERENCES tenants(id)
);

-- 功能使用统计表（每日聚合）
CREATE TABLE feature_usage_daily (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    tenant_id INTEGER NOT NULL,
    date DATE NOT NULL,
    feature_name TEXT NOT NULL,         -- 功能名称
    usage_count INTEGER DEFAULT 0,      -- 使用次数
    unique_users INTEGER DEFAULT 0,     -- 独立用户数
    FOREIGN KEY (tenant_id) REFERENCES tenants(id),
    UNIQUE(tenant_id, date, feature_name)
);

-- 客户导入统计表
CREATE TABLE import_stats (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    tenant_id INTEGER NOT NULL,
    import_date DATE NOT NULL,
    total_records INTEGER DEFAULT 0,    -- 总记录数
    successful_records INTEGER DEFAULT 0,-- 成功记录数
    failed_records INTEGER DEFAULT 0,   -- 失败记录数
    FOREIGN KEY (tenant_id) REFERENCES tenants(id),
    UNIQUE(tenant_id, import_date)
);

-- 诊断统计表
CREATE TABLE diagnosis_stats (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    tenant_id INTEGER NOT NULL,
    diagnosis_date DATE NOT NULL,
    total_diagnosis INTEGER DEFAULT 0,  -- 总诊断次数
    category_a INTEGER DEFAULT 0,       -- A类数量
    category_b INTEGER DEFAULT 0,       -- B类数量
    category_c INTEGER DEFAULT 0,       -- C类数量
    FOREIGN KEY (tenant_id) REFERENCES tenants(id),
    UNIQUE(tenant_id, diagnosis_date)
);

-- 方案生成统计表
CREATE TABLE plan_stats (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    tenant_id INTEGER NOT NULL,
    plan_date DATE NOT NULL,
    total_plans INTEGER DEFAULT 0,      -- 总方案数
    plan_type TEXT,                     -- 方案类型
    execution_status TEXT,              -- 执行状态（pending/in_progress/completed）
    FOREIGN KEY (tenant_id) REFERENCES tenants(id)
);
```

### 2.3 数据统计SQL查询

**每日活跃用户数**：
```sql
SELECT 
    tenant_id,
    DATE(timestamp) as date,
    COUNT(DISTINCT user_id) as daily_active_users
FROM usage_logs
WHERE event_type IN ('page_view', 'feature_usage')
GROUP BY tenant_id, DATE(timestamp)
ORDER BY date DESC;
```

**功能使用率Top 5**：
```sql
SELECT 
    tenant_id,
    feature_name,
    SUM(usage_count) as total_usage,
    COUNT(DISTINCT date) as days_used
FROM feature_usage_daily
WHERE date >= DATE('now', '-7 days')
GROUP BY tenant_id, feature_name
ORDER BY total_usage DESC
LIMIT 5;
```

**客户导入成功率**：
```sql
SELECT 
    tenant_id,
    AVG(successful_records * 100.0 / total_records) as success_rate
FROM import_stats
WHERE import_date >= DATE('now', '-30 days')
GROUP BY tenant_id;
```

**诊断结果分布**：
```sql
SELECT 
    tenant_id,
    SUM(category_a) as total_a,
    SUM(category_b) as total_b,
    SUM(category_c) as total_c,
    SUM(total_diagnosis) as total_diagnosis,
    ROUND(SUM(category_a) * 100.0 / SUM(total_diagnosis), 2) as a_percentage,
    ROUND(SUM(category_b) * 100.0 / SUM(total_diagnosis), 2) as b_percentage,
    ROUND(SUM(category_c) * 100.0 / SUM(total_diagnosis), 2) as c_percentage
FROM diagnosis_stats
WHERE diagnosis_date >= DATE('now', '-30 days')
GROUP BY tenant_id;
```

## 4. 数据看板设计

### 4.1 管理员数据看板

**首页概览**：
```sql
-- 获取所有朋友的汇总数据
SELECT 
    t.name as friend_name,
    COUNT(DISTINCT DATE(u.timestamp)) as active_days_last_7,
    COUNT(DISTINCT u.user_id) as active_users,
    SUM(CASE WHEN u.event_type = 'diagnosis' THEN 1 ELSE 0 END) as diagnosis_count,
    SUM(CASE WHEN u.event_type = 'plan_generation' THEN 1 ELSE 0 END) as plan_count,
    COALESCE(i.total_records, 0) as imported_customers,
    COALESCE(d.total_diagnosis, 0) as total_diagnosis,
    COALESCE(p.total_plans, 0) as total_plans
FROM tenants t
LEFT JOIN usage_logs u ON t.id = u.tenant_id 
    AND u.timestamp >= DATE('now', '-7 days')
LEFT JOIN (
    SELECT tenant_id, SUM(total_records) as total_records
    FROM import_stats
    WHERE import_date >= DATE('now', '-30 days')
    GROUP BY tenant_id
) i ON t.id = i.tenant_id
LEFT JOIN (
    SELECT tenant_id, SUM(total_diagnosis) as total_diagnosis
    FROM diagnosis_stats
    WHERE diagnosis_date >= DATE('now', '-30 days')
    GROUP BY tenant_id
) d ON t.id = d.tenant_id
LEFT JOIN (
    SELECT tenant_id, SUM(total_plans) as total_plans
    FROM plan_stats
    WHERE plan_date >= DATE('now', '-30 days')
    GROUP BY tenant_id
) p ON t.id = p.tenant_id
GROUP BY t.id, t.name
ORDER BY active_days_last_7 DESC;
```

## 5. 反馈处理流程

### 5.1 反馈分类与优先级

| 类别 | 描述 | 响应时间 | 处理人 |
|------|------|----------|--------|
| **P0** | 系统崩溃/无法使用 | 24小时内 | 开发 |
| **P1** | 核心功能问题 | 3天内 | 开发 |
| **P2** | 体验优化建议 | 1周内评估 | 产品 |
| **P3** | 新功能建议 | 记录需求池 | 产品 |

### 5.2 反馈处理工作流

```
收到反馈 → 分类处理 → 优先级排序 → 修复优化

步骤1：接收反馈
├── 来源：问卷、访谈、微信、系统反馈
├── 记录：反馈系统记录
└── 分类：P0/P1/P2/P3

步骤2：评估影响
├── 影响范围：单个朋友还是所有朋友？
├── 严重程度：影响使用还是影响体验？
├── 紧急程度：需要立即解决还是可以等等？
└── 解决成本：小修小补还是大改？

步骤3：分配处理
├── P0/P1：立即分配给开发
├── P2：产品评估后安排
└── P3：记录到需求池

步骤4：修复验证
├── 修复：开发完成修复
├── 测试：测试验证通过
└── 发布：发布到所有实例

步骤5：反馈闭环
├── 通知：通知反馈的朋友
├── 确认：确认问题解决
└── 感谢：感谢反馈
```

### 7.3 验证风险控制

**风险1：朋友反馈不真实**
- **监控指标**：使用数据 vs 口头反馈
- **应对策略**：只看数据，不只听反馈
- **验证方法**：对比使用数据和朋友说辞

**风险2：产品过重**
- **监控指标**：上手时间、功能使用率
- **应对策略**：V0聚焦10项核心功能
- **验证方法**：新朋友1小时内能否学会核心功能

**风险3：价值不明确**
- **监控指标**：成交案例、朋友推荐
- **应对策略**：3个月内必须有1单成交归因
- **验证方法**：深度访谈，量化价值

**风险4：迭代太慢**
- **监控指标**：问题修复时间、反馈响应速度
- **应对策略**：建立快速响应机制（1周内修复）
- **验证方法**：跟踪问题从反馈到修复的时间

## 10. 关键成功因素

### 10.1 数据驱动的关键

**不要相信感觉，相信数据**：
- 朋友说"好用"但不用 → 产品有问题
- 朋友说"一般"但天天用 → 产品有价值
- 数据不说话，但数据不撒谎

**关键数据指标**：
- 活跃天数 > 3天/周
- 诊断次数 > 5次/周  
- 方案生成 > 2次/周
- 成交反馈 ≥ 1单/朋友

### 10.2 朋友沟通的关键

**建立信任关系**：
- 坦诚：告诉朋友这是内测产品
- 开放：欢迎所有反馈，包括负面
- 及时：快速响应问题，快速修复
- 感恩：感谢朋友的时间和建议

**沟通技巧**：
- 问具体问题，不问模糊问题
- 量化价值，不空谈感受
- 对比前后，不主观判断
- 聚焦行为，不猜测动机

### 10.3 迭代优化的关键

**快速迭代，小步快跑**：
- 每周发布修复版本
- 每月发布功能版本
- 每季度发布大版本

**数据驱动优化**：
- 基于数据做决策
- 基于反馈改产品
- 基于结果调方向

**朋友参与优化**：
- 让朋友参与设计
- 让朋友测试新功能
- 让朋友评价改进
