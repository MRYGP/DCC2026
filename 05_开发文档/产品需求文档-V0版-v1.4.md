## 1. 产品概述

### 1.1 产品愿景

**为什么做**：
帮助贷款中介复活存量客户，提升业务效率，实现：
- 存量复活：将B/C类客户通过改造升级为A类
- 精准匹配：准确匹配客户与银行产品（减少拒贷）
- 长期陪跑：跟踪改造进度，形成长期服务关系

**V0商业目标**：
- 3-5个朋友真实使用验证
- 单个朋友能独立完成客户诊断和方案生成
- 至少1个朋友因系统成交1单
- 朋友活跃率>80%（每周登录3天以上）

**北极星指标**：
朋友真实使用数据（活跃天数、诊断次数、方案生成次数）

### 1.2 核心价值主张（朋友验证版）

**解决朋友三大痛点**：
1. **存量客户管理难**：Excel管理混乱，容易遗漏
2. **诊断依赖经验**：人工判断不准，效率低
3. **方案不可执行**：只有建议，没有具体任务清单

**我们的解决方案（V0核心）**：
1. **批量导入**：Excel一键导入，节省时间
2. **自动诊断**：六宫格体质评分，自动分层
3. **智能匹配**：匹配银行产品，识别缺口
4. **方案生成**：生成可执行改造方案
5. **自动提醒**：A/B/C分类提醒，不漏客户

### 1.3 部署形态（V0调整）

**V0采用：独立部署（每个朋友独立实例）**

**理由**：
1. **朋友更信任**：数据在自己手里，不是"公用系统"
2. **完全隔离**：一个朋友的问题不影响其他朋友
3. **品牌独立**：每个朋友有自己的Logo和域名
4. **成本可控**：5个容器，每月250元
5. **验证真实**：朋友真实使用，不是"试试看"

**V0交付物**：
- 独立部署包（Docker容器）
- 一键部署脚本
- 每个朋友独立数据库
- 独立域名访问（friendX.yourdomain.com）

**V1规划**：
- 根据验证结果决定：继续独立部署 or 转向多租户SaaS
- 如果朋友验证成功 → 扩展到30个中介
- 如果验证失败 → 深度分析，调整方向

### 1.4 成功指标（朋友验证版）

**北极星指标**：
- 朋友真实使用数据（不看口头评价，看实际行为）

**关键指标（朋友层面）**：
- ✅ 导入客户数 > 50个/朋友
- ✅ 活跃天数 > 3天/周
- ✅ 诊断次数 > 5次/周
- ✅ 方案生成次数 > 2次/周
- ✅ 成交反馈 ≥ 1单/朋友（3个月内）

**观察指标（产品层面）**：
- 批量导入成功率 > 90%
- 产品匹配有结果率 > 80%
- 方案生成完成率 100%
- 朋友满意度（NPS） > 70%

## 2. 用户分析（朋友验证版）

### 2.1 用户角色（V0版本）

**主要角色：中介朋友**（3-5个信任的朋友）
- 权限：完整系统权限
- 登录方式：账号密码
- 特点：愿意真实使用，愿意给真实反馈

**次要角色（V0不考虑）**：
- ❌ 系统管理员（内部）：V0阶段手动运维
- ❌ 企业主（外部用户）：V1再考虑
- ❌ 多租户管理员：V0用独立部署替代

### 2.2 用户旅程地图（朋友验证版）

```
flowchart TD
    A[朋友拿到系统] --> B[导入历史客户]
    B --> C{自动诊断分层}
    C -->|A类| D[立即匹配银行产品]
    C -->|B类| E[生成改造方案]
    C -->|C类| F[存档+6个月后复评]
    D --> G[跟进进件]
    E --> H[执行改造任务]
    H --> I[3个月后重新诊断]
    F --> J[定期提醒复评]
    
    G & I & J --> K[结果回填]
    K --> L[数据飞轮启动]
```

### 2.3 核心业务流程（朋友验证版）

**流程1：朋友首次使用**
1. 朋友收到部署链接和账号
2. 登录系统，配置Logo和品牌
3. 批量导入历史客户数据（Excel）
4. 系统自动诊断分层（A/B/C）
5. 查看诊断结果，开始使用

**流程2：日常使用循环**
1. 导入新客户数据
2. 自动诊断分层
3. A类：立即匹配产品，跟进进件
4. B类：生成改造方案，执行任务
5. C类：存档，设置复评提醒
6. 结果回填，积累数据

## 3. V0功能需求（10项核心功能）

> 基于朋友验证路径重新定义，聚焦朋友真正需要的核心功能

### 3.1 核心业务功能（6项功能）

#### 功能1：批量导入Excel（P0）
- **功能描述**：上传包含客户信息的Excel，批量导入系统
- **输入**：Excel文件（.xlsx格式）
- **13个必填字段（最终字典）**：
  1. 企业名称（文本）
  2. 注册资本（数值，单位万元）
  3. 成立时间（日期，格式YYYY-MM-DD）
  4. 法人姓名（文本）
  5. 年开票（数值，单位万元）
  6. 年纳税（数值，单位万元）
  7. 对公流水月均（数值，单位万元）
  8. 法人征信（文本，如"无逾期"/"M1逾期"/"M3+逾期"）
  9. 企业征信（文本，如"正常"/"异常"）
  10. 主营业务（文本）
  11. 行业（文本，如"制造业"/"服务业"）
  12. 贷款需求（数值，单位万元）
  13. 抵押物情况（文本，如"有房产/150万"/"无"）
- **简化点**：
  - 只做必填字段验证，不做复杂格式校验
  - 错误提示简单明了
  - 支持100条数据5秒内完成
- **验收标准**：能上传、能解析、能导入、错误信息清晰

#### 功能2：自动诊断分层（P0）
- **功能描述**：根据六宫格体质评分，自动分为A/B/C三类
- **算法**：
  - 流水体质（200分）
  - 票税体质（200分）
  - 征信体质（200分）
  - 资产体质（150分）
  - 资质体质（150分）
  - 行业体质（100分）
- **分层规则**：
  - A类（总分≥600）：现在能贷
  - B类（400≤总分<600）：值得改造
  - C类（总分<400）：暂不跟进
- **输出**：分层结果、评分明细、分层理由
- **验收标准**：分层准确、有评分明细、可解释

#### 功能3：产品匹配（P0）
- **功能描述**：匹配20个银行产品，推荐最适合的
- **匹配逻辑**：
  1. 遍历20个银行产品准入条件
  2. 逐条对比客户条件
  3. 识别具体差距（数值型差距需量化）
  4. 输出匹配结果+差距清单
- **简化点**：
  - 先匹配5个最常用产品（覆盖80%场景）
  - 详细差距分析可简化
  - 推荐理由简单明了
- **输出**：匹配产品列表、推荐理由、差距清单
- **验收标准**：匹配准确、理由清晰、差距可理解

#### 功能4：方案生成（P0）
- **功能描述**：生成结构化改造方案包
- **输入**：客户缺口数据
- **输出**：含5个标准部分的方案包：
  1. 缺口清单（差什么、差多少）
  2. 任务清单（做什么、谁负责、证据要求、完成标准）
  3. 推荐服务商（2-3家可选，或中介自己找）
  4. 效果预览（改造后预计能贷多少）⚠️ 以银行审批为准
  5. 成本估算（改造成本、ROI）
- **简化点**：
  - 从12个方案库匹配简化为3个核心方案
  - ROI计算可简化
  - 服务商推荐可简化
- **验收标准**：方案结构完整、可执行、有免责声明

#### 功能5：自动提醒（P1）
- **功能描述**：根据不同客户类型设置自动提醒
- **触发条件**：
  - A类：立即提醒（可跟进进件）
  - B类：到期提醒（3个月改造期到期）
  - C类：定期复评提醒（6个月后）
- **输出**：系统内提醒（先不做邮件/微信通知）
- **简化点**：
  - 只做系统内提醒
  - 邮件/微信通知V0.1再做
- **验收标准**：按时触发提醒、提醒准确

#### 功能6：可解释+人工复核（P1）
- **功能描述**：提供可解释结果并支持人工复核
- **算法护栏**：
  1. 可解释：每个分层都能说清楚为什么（基于哪些字段、哪些规则）
  2. 人工复核：中介可以改判系统的分层（A→B或B→A）
  3. 学习闭环：系统记录人工修正理由，用于后续优化
- **验收标准**：分层理由清晰、支持人工修正、保留修正记录

#### 功能10：微信服务号推送（P0）⭐⭐⭐⭐⭐

**功能描述**：通过微信服务号推送银行政策更新、客户状态变化提醒

**触发频率**：
- **政策更新**：每周2-3次（周一、周三、周五早9点）
- **客户提醒**：每天1次（早上9:00）
- **竞争压力**：每周1次（周五下午，可选）

**消息模板**：

**模板1：政策更新**
```
【政策更新】XX银行-XX产品
准入条件变化：XXX
影响您的客户：X个
点击查看 → http://friend1.yourdomain.com/updates/123
```

**模板2：客户提醒**
```
【客户提醒】您有X个客户可立即跟进
• 张三 - 流水已满足浦发银行条件，预计可贷50万
• 李四 - 高新认定已完成，B类→A类
→ 点击查看详情并联系客户
```

**点击后行为**：直接跳转到H5对应页面；受影响客户清单（已筛选好）；待办事项（可一键操作）

**技术实现**：微信服务号模板消息API；定时任务（Celery或APScheduler）；客户筛选逻辑（匹配政策变化）

**数据库表**：
```sql
CREATE TABLE push_notifications (
    id INTEGER PRIMARY KEY,
    partner_id TEXT,
    type TEXT, -- 'policy_update' | 'customer_reminder' | 'social_proof'
    title TEXT,
    content TEXT,
    target_url TEXT,
    sent_at TIMESTAMP,
    opened_at TIMESTAMP,
    clicked_at TIMESTAMP
);
```

**验收标准**：推送成功率 > 95%；打开率 > 60%（政策更新）/ > 70%（客户提醒）；点击率 > 80%；延迟 < 5秒

**风险**：微信服务号审核需要5-7天（需提前申请）；降级方案：先用系统内通知（红点）

#### 功能11：首页待办清单（P0）⭐⭐⭐⭐⭐

**功能描述**：首页展示今日待办事项，而非数据看板

**设计原则**：首页就是行动清单；每个待办可一键操作；数字具体化（"3个A类客户"）；颜色编码（红黄绿）

**展示内容**：
```
┌─────────────────────────────────────┐
│ 今日待办                              │
├─────────────────────────────────────┤
│ 🔴 3个A类客户可立即跟进               │

│ 🟢 2个新客户等待诊断                  │
│    → 开始诊断                         │
├─────────────────────────────────────┤
```

**交互设计**：点击"查看详情"→客户列表（已筛选）；点击"查看任务清单"→改造任务页；点击"开始诊断"→诊断页面

**技术实现**：React组件（首页重新设计）；实时数据查询（FastAPI接口）；缓存优化（Redis，V1再做）

**验收标准**：首页加载时间 < 1秒；待办点击率 > 80%；数据准确率 > 99%；用户满意度 > 90%

#### 功能12：一键操作（P0）⭐⭐⭐⭐⭐

**功能描述**：降低用户行动门槛，从3步变1步

**场景1（政策更新后）**：优化后 2 步——点击提醒→看受影响客户清单；点击客户→一键跳转微信预填文案

**场景2（客户提醒后）**：优化后 2 步——点击提醒；点击"发送微信"→跳转微信（预填文案）→发送

**预填文案模板**：张总您好，我是XX（中介名称）；您公司现在满足浦发银行的贷款条件：最高可贷50万、年化利率4.5%、审批周期3-5个工作日；方便聊聊吗？

**技术实现**：H5深度链接（跳转微信）；URL Scheme：`weixin://dl/business/?t=XXX`；文案模板引擎（Jinja2）；客户信息变量替换

**验收标准**：跳转成功率 > 90%；操作路径 ≤ 2步；完成率 > 70%；平均完成时间 < 30秒

**技术风险**：微信跳转在iOS和Android表现可能不一致，需真机测试

**验收标准**：反馈延迟 < 1秒；反馈准确率 > 95%；弹窗显示时长3秒；用户满意度 > 90%

**功能描述**：首页展示用户数据价值，增加退出成本

**设计原则**：让用户看到"用得越多越值钱"；数据准确率 > 99%；实时更新（延迟 < 5秒）

**展示内容**：管理客户数、诊断次数、成交记录、数据完整度；提示"数据完整度越高，系统推荐越准确"；行动引导"→ 完善客户资料（还差15%）"

**数据完整度计算**：13个必填字段，完整度 = (已填字段数/13)×100%

**技术实现**：统计查询SQL；实时更新（WebSocket or 轮询5秒）；缓存优化（V1再做）

**验收标准**：数据准确率 > 99%；实时更新延迟 < 5秒；查看率 > 70%；点击"完善客户资料"转化率 > 40%

## 5. 业务规则（保持不变）

### 5.1 诊断规则

**六宫格体质评分模型**：
| 体质维度 | 核心指标 | 满分 | 权重说明 |
|---------|---------|------|----------|
| 流水体质 | 对公流水规模、稳定性 | 200分 | 银行最看重连续现金流 |
| 票税体质 | 年开票额、年纳税额 | 200分 | 税贷/发票贷核心依据 |
| 征信体质 | 查询次数、负债结构 | 200分 | 一票否决项（严重逾期） |
| 资产体质 | 抵押物价值、保证人 | 150分 | 增信手段，非必需 |
| 资质体质 | 高新/专精特新认定 | 150分 | 政策优惠项 |
| 行业体质 | 行业风险、政策支持 | 100分 | 负面筛查维度 |

**分层阈值**：
- A类（立即可贷）：总分 ≥ 600分
- B类（值得改造）：400分 ≤ 总分 < 600分
- C类（暂不跟进）：总分 < 400分

### 5.2 匹配规则

**产品匹配逻辑**：
1. 遍历20个银行产品准入条件
2. 逐条对比客户条件
3. 识别具体差距（数值型差距需量化）
4. 输出匹配结果+差距清单

**改造方案匹配**：
- 从12个标准方案库中匹配
- 基于具体差距类型（如开票不足→开票提升术）
- 推荐2-3个可行性方案

### 5.3 合规规则（简化版）

**禁用表述**：
- ❌ "低息快贷"
- ❌ "无条件放款"
- ❌ "百分百放款"
- ❌ "内部渠道"

**统一措辞**：
- ✅ "初步评估/预估/建议"
- ✅ "以银行审批为准"
- ✅ "仅供参考"
- ✅ "改造后预计可贷"

## 8. 风险与依赖（朋友验证版）

### 8.1 技术风险

**主要风险**：
1. **独立部署复杂度**：朋友IT能力参差不齐
2. **SQLite性能限制**：数据量大时可能需优化
3. **一键部署脚本稳定性**：不同环境兼容性

**缓解措施**：
- 简化部署流程（一键脚本）
- 提供详细部署文档 + 视频教程
- 7x24小时技术支持

### 8.2 业务风险

**主要风险**：
1. **朋友不真实使用**：出于情面说"好用"但不真用
2. **产品过重**：功能太多，朋友学不会
3. **价值不明确**：朋友说不清"有什么用"

**缓解措施**：
- 不看口头反馈，看使用数据（活跃天数、诊断次数）
- V0聚焦10项核心功能，去掉复杂的
- 量化价值：3个月内必须有1单成交归因

### 8.3 依赖项

**技术依赖**：
- Claude Code：AI辅助开发
- 现有知识库：银行政策库+改造方案库
- 开源组件：FastAPI、React、Ant Design、Docker

**业务依赖**：
- 朋友配合：3-5个朋友愿意真实使用
- 真实数据：朋友提供50+真实客户数据
- 真实反馈：朋友给真实使用反馈

## 9. 验证指标与成功标准

### 9.1 朋友使用指标（客观数据）

| 指标 | 定义 | 目标值 | 验证方式 |
|------|------|--------|----------|
| **导入客户数** | 每个朋友实际导入的客户数量 | > 50个/朋友 | 系统统计 |
| **活跃天数** | 每周登录使用的天数 | > 3天/周 | 系统统计 |
| **诊断次数** | 使用诊断功能的次数 | > 5次/周 | 系统统计 |
| **方案生成次数** | 生成改造方案的次数 | > 2次/周 | 系统统计 |
| **成交反馈** | 系统帮助成交的订单数 | ≥ 1单/朋友 | 人工访谈 |

### 9.2 朋友反馈指标（主观评价）

**每周轻量问卷**：
1. "这周你用了系统几次？"
2. "哪个功能最有用？"
3. "哪个功能最难用？"
4. "有没有遇到bug或卡点？"

**每月深度访谈**：
1. "用了系统后，你的工作有什么改变？"
2. "如果没有这个系统，你会怎么做？"
3. "你会推荐给其他中介朋友吗？"
4. "你愿意为这个系统付多少钱？"

### 9.3 第13周复盘标准

**成功标准（全部满足）**：
- ✅ 5个朋友都在持续使用（活跃率>80%）
- ✅ 至少2个朋友明确说"很有用"
- ✅ 至少1个朋友因系统成交1单
- ✅ 建立了快速响应机制（1周内修复问题）

**失败信号（任一出现）**：
- ❌ 活跃率<50%（朋友没真用）
- ❌ 0个朋友说"很有用"（产品无价值）
- ❌ 0单成交（商业价值存疑）
- ❌ 响应时间>2周（迭代太慢）
