## 这个岗位的核心工作是什么

### 一句话版本

把银行的贷款产品翻译成机器能读的配置文件（我们叫 Skill），然后写程序让这些配置跑起来，自动判断「这个客户能不能贷、能贷多少、差在哪里」。

### 展开说

银行有几十个贷款产品，每个产品有不同的准入条件（流水要多少、纳税要多少、征信不能太差）。中介现在靠经验和记忆来匹配客户和产品——效率低、容易出错。

我们要做的是：
1. 把每个银行产品的规则写成一个 Skill 配置文件（YAML 格式）
2. 写一段程序，拿客户数据去逐个比对所有 Skill
3. 输出一份报告：哪些产品能贷、预估多少额度、不符合的差在哪里

**你的主要工作 = 设计 Skill + 验证输出**。写代码是让 Skill 跑起来的手段，不是目的。

### 具体例子

假设工商银行有一个「税贷」产品，准入条件是：月流水≥10 万、年纳税≥1 万、近 6 月征信查询≤6 次。

传统做法（硬编码）：
```python
def check_icbc_tax_loan(customer):
    if customer.monthly_revenue < 100000:
        return False, "月流水不足10万"
    if customer.annual_tax < 10000:
        return False, "年纳税不足1万"
    if customer.credit_query_count > 6:
        return False, "征信查询超过6次"
    return True, f"预估额度{customer.annual_tax * 25}元"
```

我们的做法（Skill 配置，符合《Skill 格式规范》最小可运行结构）：

```yaml
metadata:
  skill_id: "PRODUCT_ICBC_TAX_001"
  skill_name: "工商银行税贷"
  skill_type: "product"
  source: "工商银行官网"
  verified_date: "2026-02-01"
  verified_by: "产品"
  region: "全国"
  version: "1.0.0"
  created_by: "admin@dcc2026.com"
  status: "active"

eligibility:
  logic: "all"
  conditions:
    - field: "monthly_revenue"
      operator: ">="
      value: 100000
      description: "月均流水≥10万元"
    - field: "annual_tax"
      operator: ">="
      value: 10000
      description: "年纳税≥1万元"
    - field: "credit_query_6m"
      operator: "<="
      value: 6
      description: "近6月征信查询≤6次"

amount_calculation:
  formula_type: "expression"
  formula: "annual_tax * 25"
  amount_range: { min: 0, max: 5000000, unit: "元" }
```

**区别在哪**：
- 硬编码：改一个条件要改代码、重新测试、重新部署
- Skill 配置：改 YAML 即可，不动代码，非技术人员也能看懂规则，且程序能自动生成客户看得懂的报告

**这就是你的核心能力**：不是写更多代码，而是设计更好的规则，让一段通用程序能处理几十个不同产品。

### 在行业里这叫什么

类似概念：LangChain 的 Agent 配置、Anthropic 的 MCP、OpenAI 的 GPTs——用配置控制行为，而不是硬编码。我们在贷款领域的应用是首创，方法论可迁移到其他 AI 应用。

**具体安排**：

- **Day 4–Week 2**：优先全周长沙线下（培训期，面对面效率更高）
- **Week 3–8**：每周至少 3 天长沙线下，其余时间可远程

**弹性处理**：
- 毕业论文/答辩周：提前 1 周告知，可协商豁免，不影响验收节点即可
- 临时家事：提前协调，优先保障
- 确实无法来长沙的：建议等我们开放纯远程岗位后再申请

**Q4：每周需要投入多少时间？你们更看重总时长还是连续深度？**

**A：总时长是基线，连续深度时段是真正的要求。**

我们更看重每天能有连续 2–3 小时的专注时间，而不是碎片化凑满总时长。Skill 设计和调试需要进入状态，半小时切一次根本做不完整。

**全职实习生**：
- Week 1–2：每周 25–30 小时（学习期，弹性安排）
- Week 3–8：每周 30–35 小时（开发期）
- Week 7–8：关键验收周，需要集中投入（提前 1 周通知）

**非全职实习生**：
- Week 1–2：每周 15–20 小时
- Week 3–8：每周 20–25 小时
- 硬要求：每天至少有连续 2–3 小时专注时段（碎片化等于没投入）

**弹性安排**：
- 不强制打卡，只要保证任务进度
- 家里有事优先，提前协调即可

**如果我是大四学生，有课怎么办**：
- 建议：寒假/暑假全职实习
- 或者：学期内非全职（但每天必须保证连续时段）

**Q4.5：Week 1–2 结束，第一个 Skill 要做到什么程度才算过关？**

**A：能跑通不算完成，过关标准如下：**

- 准入条件字段齐全，单位统一为「元」（不能混用万元）
- 至少 2 个 pytest：一个符合、一个不符合，建议覆盖「等于阈值」的边界情况
- 输出结果必须包含：符合/不符合 + 原因说明 + 预估额度
- 有简单的 README 说明怎么跑

Week 1 培训时会提供示范 Skill；若想提前了解格式，可查看仓库示例：`01_银行政策库/EXAMPLE_SKILL_虚构税贷_招聘示例.yaml`。你照着格式来，规则内容需要自己理解和填写。Skill 的银行产品规则由我们提供原始资料，有歧义时由创始人裁决。

- ✅ 如果你不愿意，我们帮你推荐其他机会（创业圈人脉）
- ✅ 2 个月的项目经历和代码，你都可以带走（不会「白干」）

**对你的保障**：
- 你跟的不是「一个可能倒闭的项目」

## Step 2：小作业（约 30–60 分钟）

**目的**：验证「能不能把规则写成配置 + 能不能自证正确」（会写测试、单位清晰、能解释不符合原因），而不是背八股。

通过 Step 1 后，我们会发小作业题目。请你按**收到题目后 3 个工作日内**提交（逾期未交视为放弃）。交付方式：GitHub repo 或 zip，通过邮件发链接给我们。

### 后端/全栈小作业（概要，以实际发出的题目为准）

**任务**：
1. 写一个虚构银行产品「税贷」的 Skill（YAML），准入条件：
   - 月流水≥10 万元（即 100000 元）
   - 年纳税≥1 万元（即 10000 元）
   - 近 6 月征信查询≤6 次
   - 额度（元）= 年纳税（元）× 25
2. 写 Python 函数：输入客户 dict + Skill 文件，输出「符合/不符合 + 原因 + 预估额度」
3. 写 2 个 pytest：一个符合、一个不符合（建议覆盖边界：等于阈值）
4. **所有金额字段单位统一为「元」**，在 Skill 与代码里写清单位

### 前端赛道小作业（概要）

**任务**：
1. 用 React + Ant Design 做一个表单：月流水（元）、年纳税（元）、征信查询次数
2. 校验：必填、数字、≥0
3. 提交后展示是否符合**与后端同一套**税贷准入条件：
   - 月流水≥10 万元（即 100000 元）
   - 年纳税≥1 万元（即 10000 元）
   - 近 6 月征信查询≤6 次

**说明**：全栈方向可做后端为主，前端为加分项；若同时提交前端小作业，可在面试中加分，不做也不影响进入面试。

### 提交要求（必读）

- **README**：怎么跑、怎么测、你花了多久（如实填写，我们会结合代码量与复杂度交叉看，不诚信会一票否决）
- 若使用 AI：写清楚「AI 帮了什么 + 你怎么验证正确性」
- 交付：GitHub repo 或 zip，链接发回我们邮箱

**我们怎么判定**：
- ✅ 单位写清楚 + 有测试 + 能解释「为什么不符合」→ 进入 Step 3
- ⚠️ 只有功能跑通、无测试或单位混乱 → 可能要求补测一次再决定
- ❌ 明显复制粘贴且解释不清 → 不进入下一步

## Step 3：线上面试（约 30–40 分钟）

**目的**：确认真实性、思维方式与可合作度。

**结构建议**（约 30–40 分钟，可根据情况微调）：
- **约 5 分钟**：自我介绍（我们会追问你项目里最关键的一件事）
- **约 15 分钟**：围绕小作业深挖——为什么这么设计、边界怎么测、哪里可能出错
- **约 10 分钟**：DCC 特色题至少 1 道（从下面 3 题中选）
- **约 5 分钟**：时间承诺核验（8 周、每周 25–30h 或你 Step 1 填写的承诺；若与 Step 1 不一致，请说明原因）

**DCC 特色题（必问至少 1 道）**：
1. **Skill 理解**：为什么用 YAML 配置而不是硬编码 if-else？好处和麻烦分别是什么？
2. **单位陷阱**：元/万元混用会发生什么？你会怎么在系统层面防住？
3. **「没有测试 = 未交付」**：你怎么看？你现在的交付标准是什么？

**通过 Step 3 之后**：
- 我们会在 **1–2 个工作日内** 发 offer
- 入职时间、实习协议等另行沟通
