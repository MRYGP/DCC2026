## 二、我们是谁 / 我们不是谁

### 2.1 定位与角色

**我们是**：
- ✅ SaaS技术提供者（白标配置化基座）
- ✅ 方案提供者（任务清单+证据要求+推荐服务商+效果预览）

- ❌ 结果承诺者（不承诺通过/额度/利率）

### 2.2 边界清晰（对比表）

| 维度 | 我们是 | 我们不是 |

## 三、核心价值主张

### 3.1 目标用户的真实困境

**困境1：存量客户睡在Excel，记不住谁该联系**
- 每个中介手里有100-200个"以前没成交"的历史客户
- 这些客户：知根知底、有联系方式、有明确需求
- 但中介记不住：谁现在条件够了？谁还差多久？谁该主动联系？
- 结果：大部分客户就这样流失了

**困境2：新客户获客成本高、转化率低**
- 获取新客户成本：3000-5000元/个
- 新客户转化率：5-10%
- 很多"新客户"其实是"别人的前女友"（已被其他中介接触过）

**困境3：诊断靠经验，不够精准**
- 靠中介个人经验判断"能不能贷"
- 容易漏掉匹配的产品
- 容易给客户错误的建议
- 客户失败后，不知道为什么失败

**困境4：给了改造建议，客户不执行**
- 中介："你开票不够，需要提升"
- 客户："具体怎么做？找谁做？成本多少？"
- 中介：说不清楚 → 客户不执行 → 机会流失

**困境5：不知道去哪找服务商，服务商质量参差不齐**
- 客户需要改造时，不知道找谁
- 网上搜索，几十家机构，不知道谁靠谱
- 担心被骗、被坑、被宰

### 3.2 我们的解决方案（三大引擎）

| 引擎 | 解决什么 | 如何做 | 价值 |
|------|----------|--------|------|
| **存量复活引擎** | 激活睡在Excel的200个"前女友" | 批量导入→自动分层（A/B/C）→自动提醒（到期该联系谁） | 20%激活率 = 40个客户，零获客成本 |
| **精准匹配引擎** | 把"能不能贷"变成可解释的缺口+改造方案 | 诊断→产品匹配→缺口识别→改造方案（任务清单+推荐服务商） | 匹配准确率>85%，减少返工 |
| **长期陪跑引擎** | 客户终身价值最大化 | 所有客户都给改造方案（成功也给、失败也给），持续陪跑 | 中介收入提升2.7倍 |

**核心创新点**：
1. **存量复活**：把睡在Excel的客户变成可管理的资产（市场上少有）
2. **方案任务化**：不是给"建议报告"，而是给"可执行任务清单+证据+推荐服务商"
3. **认证服务商网络**：解决"不知道去哪找、谁靠谱"的痛点
4. **白标模式**：每个中介独立产品，数据归自己，不担心被平台抢客户

### 3.3 与竞品的核心差异

| 维度 | 传统助贷CRM | 我们 |
|------|-----------|------|
| **核心定位** | 线索管理+进件效率 | 存量复活+改造执行 |
| **主要场景** | 新客户获取 | 历史客户激活 |
| **核心功能** | 跟单管理 | 批量导入+自动分层+自动提醒 |
| **改造服务** | 无 | 认证服务商网络 |
| **产品形态** | 标准SaaS | 白标定制 |
| **数据归属** | 平台 | 中介公司 |

**我们更聚焦**：
- 聚焦"存量客户复活"而非"新客户效率"
- 聚焦"改造方案执行"而非"线索管理"
- 聚焦"可验收的任务清单"而非"建议报告"

## 四、服务逻辑

### 4.1 统一服务逻辑：所有客户都给改造方案

| 客户类型 | 当前状态 | 给什么 | 为什么 |
|---------|---------|--------|--------|
| **A类-新客已成功** | 能贷80万 | 改造方案（3月后贷150万） | 长期陪跑、客户终身价值最大化 |
| **B类-新客失败** | 贷不了 | 改造方案（3月后能贷） | 失败→成功，变废为宝 |
| **C类-老客失败过** | 睡在Excel | 改造方案（条件够了/改造后能贷） | 复活存量，零获客成本 |
| **D类-老客成功过** | 贷过50万 | 改造方案（下次贷100万） | 增量挖掘，持续创造价值 |

**核心理念**：
> 不管能不能贷、贷多少，都给改造方案。
> 
> 因为"改造"是长期陪跑的核心，是持续创造价值的抓手。
> 
> 把客户牢牢掌握在手里，做企业主的长期顾问，而不是做一锤子买卖。

## 五、商业模式

### 5.1 收入结构（第一阶段）

| 收入来源 | 金额 | 占比 | 说明 |
|---------|------|------|------|
| **SaaS订阅费** | 3.6万/中介/年 | 11% | 白标产品+所有功能 |
| **服务商返佣** | 约31万/中介/年 | 89% | 仅限我们推荐的服务商 |
| **合计** | 约34.6万/中介/年 | 100% | 单个中介年贡献 |

**规模化收入预测**：

| 阶段 | 中介数量 | 年收入 | 说明 |
|------|---------|--------|------|
| 6个月 | 30家 | 1038万 | 初步验证 |
| 12个月 | 100家 | 3460万 | 区域覆盖 |
| 24个月 | 300家 | 1.04亿 | 全国扩张 |

### 5.2 服务商返佣逻辑（关键）

**情况1：我们推荐的服务商**（核心收入）
```
客户改造费8万 → 服务商返佣1.6万（20%）
→ 我们50%（0.8万）+ 中介50%（0.8万）
```

**前提条件**：
- ✅ 对中介披露："本推荐服务商可能产生返佣分配；返佣比例：服务商返佣的50%；您可自选服务商"
- ✅ 对企业主披露："您可自选服务商，也可使用推荐服务商；推荐服务商已经过认证"
- ✅ 系统内强制勾选（不勾选无法推荐）

**情况2：中介自己找的服务商**
```
客户改造费8万 → 服务商可能给中介返佣
→ 我们不管、不收（100%归中介）
```

**为什么这样设计？**
- 我们推荐的服务商，我们提供了价值（筛选认证、智能推荐、质量监督）→ 分佣
- 中介自己找的，我们没参与 → 不收
- 中介有选择权：用我们推荐的（省心）或自己找（全拿）

### 5.3 披露与对账机制（硬规则）

**披露机制**（写进系统，强制勾选）：
- 对中介：明确告知返佣比例
- 对企业主：明确告知可自选服务商
- 不披露 = 不推荐

**对账机制**：
- 订单号、金额、服务完成定义
- 完成定义：服务商提交验收报告+中介确认
- 退款/争议处理机制
- 系统内可追溯

## 七、三大硬规则

### 7.1 白标配置边界（避免外包地狱）

**可配置（标准服务，所有中介都能自己设置）**：
```
✅ Logo、主色、域名
✅ 公司名称、联系方式
✅ 产品库启停（勾选展示哪些产品）
✅ 方案库启停（勾选展示哪些方案）
✅ 自定义字段（≤5个）
✅ 话术模板（可编辑）
```

**不定制（硬规则，超出即产品化）**：
```
❌ 任何流程/算法改动
→ 只能进"公共产品迭代池"
→ 不做单客户专属改造
```

**为什么？**
> 白标SaaS = 同一套基座、不同配置
> 
> 不是 = 同一套人力、不同定制
> 
> 如果超出配置边界，要么拒绝，要么产品化（所有客户都能用）。

### 7.2 数据权属机制（信任底座）

**数据权属**：
- 中介公司：数据控制方/处理者
- 我们：受托处理方

**三个可验收机制**：

**机制1：租户隔离+操作留痕**
- 不同中介的数据完全隔离（物理或逻辑）
- 所有操作可审计（谁导入/查看/导出/删除，全记录）
- 审计日志至少保留1年

**机制2：随时可导出/删除**
- 中介随时可导出所有数据（Excel/JSON）
- 终止合作可要求删除所有数据
- 7个工作日内完成删除并提供证明

**机制3：最小化采集+授权留档**
- 只收集13个必要字段
- 批量导入时要有"客户授权证明"
- 每个字段都说清楚用来干什么

**委托处理协议**（V0必须有）：
- 明确双方权利义务
- 明确数据使用范围
- 明确违约责任

### 7.3 合规与信任（P0级）

**合规表述规范**：

禁止表述：
- ❌ "低息快贷"
- ❌ "无条件放款"
- ❌ "百分百放款"
- ❌ "内部渠道"

统一措辞：
- ✅ "初步评估/预估/建议"
- ✅ "以银行审批为准"
- ✅ "仅供参考"

**算法护栏**：

护栏1：可解释
- 每个分层都能说清楚为什么（基于哪些字段、哪些规则）
- 每个缺口都能量化（差多少、影响什么）

护栏2：人工复核
- 中介可以改判系统的分层（A→B或B→A）
- 改判时要记录理由
- 系统学习人工修正

护栏3：免责声明
- 固化在系统里（每个评估报告都带）
- 明确"不承诺通过/额度/利率"
- 明确"以银行审批为准"

## 附录A：核心话术（长期陪跑）

### A类客户话术（已经能贷，为什么还要改造）

```
"王总，恭喜您，现在条件符合，能贷到80万。

我给您两个方案：

1. 现在立即申请80万（着急用钱）
2. 做个改造，3个月后申请150万（多拿70万）

我的建议：
如果不是特别着急，建议等3个月。
因为做了改造，不仅这次能多贷70万，
以后您的企业条件更好，贷款会越来越容易。

我是您的长期顾问，不是做一锤子买卖。
我希望帮您把企业体质养好，
以后需要钱，随时能贷到。

要不要试试第2个方案？"
```

**核心逻辑**：
- 给选择（不强迫）
- 讲长期价值（体质好 → 持续能贷）
- 定位长期顾问（不是一次性服务）

## 附录B：验证计划

### B.1 如何验证基座可用？

**验证方式**：
1. 找3个中介深度测试
2. 每个中介导入5-10个真实历史客户
3. 让中介全流程操作一遍：
   - 批量导入客户
   - 查看诊断报告
   - 选择改造方案
   - 跟进A类客户
   - 设置B类客户提醒
4. 收集反馈

**关键验证点**：
- 批量导入是否真的有用？
- 自动分层（A/B/C）是否准确？
- 改造方案是否真的可执行？
- 自动提醒是否真的有价值？

### B.2 成功标准

**最低标准**：
- 3个中介都说"批量导入有用"
- 至少2个愿意导入真实历史客户
- 至少1个愿意付费试用（哪怕是199元/月）

**理想标准**：
- 3个中介都愿意导入历史客户
- 3个中介都愿意付费
- 至少1个愿意签年费合同
- 至少1个成功激活了存量客户

**失败退出机制**：
- 如果3个中介都说"没用" → Kill项目
- 如果2个说"有用但不付费" → Pivot定价或功能
- 如果1个说"有用愿意付费" → 继续迭代
