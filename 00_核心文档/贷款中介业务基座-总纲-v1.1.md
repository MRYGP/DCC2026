## 一、一句话定位（电梯演讲）

> **给贷款中介提供白标SaaS基座，把小微融资从"碰运气的申请"变成"可解释的差距→可执行的任务→可复评的进步"。**

**核心能力**：
```
建档（数据）→ 诊断（差距）→ 匹配（当下可做）→ 方案（任务+证据）→ 提醒（时间）→ 复评（闭环）
```

**三个关键词**：
- **存量复活**：把睡在Excel的200个"前女友"变成可分层、可推进、可复评的资产
- **精准匹配**：把"能不能贷"变成可解释的缺口+任务清单
- **长期陪跑**：所有客户都给改造方案，客户终身价值最大化

**边界**：我们提供方案，第三方执行；我们是配置化基座，不做个性化定制。

### 1.5 为什么贷款中介存在（理论支撑）⭐

**市场现实**：

银行面对小微企业时，存在突出的信息不对称问题：
- 很难低成本拿到足够可信的经营信息
- 尽调成本高、风险难以评估
- 审批流程标准化，但小微企业情况复杂

**中介的价值**（经典金融中介理论）：

- **筛选**：从大量申请中筛出可做的
- **监督**：跟踪企业经营状况
- **打包**：把复杂信息变成标准材料
- **规模化尽调**：delegated monitoring（委托监督理论）

**本质上，中介在卖**：
> "把银行看不懂/不想看/看不起的小微，变成'可被审批流程消费'的材料与结构。"

**行业现状与我们的机会**：

- 灰产很多，监管经常做风险提示（"别信中介承诺/谨防诈骗/谨防高额收费"）
- **对我们是利好**：做"合规、可解释、可留痕、可对账"的系统
- 我们是在给中介提供"洗手上岸"的工具

**我们的定位**：
> 不是取代中介，而是让中介更高效、更合规、更可持续。

## 二、我们是谁 / 我们不是谁

### 2.1 定位与角色

**我们是**：
- ✅ SaaS技术提供者（白标配置化基座）
- ✅ 方案提供者（任务清单+证据要求+推荐服务商+效果预览）
- ✅ 平台推荐者（认证服务商网络）

**我们不是**：
- ❌ 改造服务交付者（不做H业务/高新认定等）
- ❌ 结果承诺者（不承诺通过/额度/利率）
- ❌ 定制开发外包（只做配置，不改流程/算法）

### 2.2 边界清晰（对比表）

| 维度 | 我们是 | 我们不是 |
|------|--------|----------|
| **定位** | 白标SaaS技术提供者 | 贷款平台/中介公司 |
| **角色** | 基座提供者+方案推荐者 | 服务执行者+结果承诺者 |
| **交付** | 系统（基座+配置）+方案（任务清单+推荐服务商） | 改造执行（H业务/认证等） |

**关键理念**：
> 我们是"配置化基座"，不是"定制化外包"；
> 我们是"方案提供者"，不是"服务交付者"；
> 我们帮中介"把客户变成资产"，不是"帮中介做业务"。

## 三、核心价值主张

### 3.1 目标用户的真实困境

**困境1：存量客户睡在Excel，记不住谁该联系**
- 每个中介手里有100-200个"以前没成交"的历史客户
- 这些客户：知根知底、有联系方式、有明确需求
- 但中介记不住：谁现在条件够了？谁还差多久？谁该主动联系？
- 结果：大部分客户就这样流失了

**困境2：新客户获客成本高、转化率低**
- 获取新客户成本：3000-5000元/个
- 新客户转化率：5-10%
- 很多"新客户"其实是"别人的前女友"（已被其他中介接触过）

**困境3：诊断靠经验，不够精准**
- 靠中介个人经验判断"能不能贷"
- 容易漏掉匹配的产品
- 容易给客户错误的建议
- 客户失败后，不知道为什么失败

**困境4：给了改造建议，客户不执行**
- 中介："你开票不够，需要提升"
- 客户："具体怎么做？找谁做？成本多少？"
- 中介：说不清楚 → 客户不执行 → 机会流失

**困境5：不知道去哪找服务商，服务商质量参差不齐**
- 客户需要改造时，不知道找谁
- 网上搜索，几十家机构，不知道谁靠谱
- 担心被骗、被坑、被宰

### 3.2 我们的解决方案（三大引擎）

| 引擎 | 解决什么 | 如何做 | 价值 |
|------|----------|--------|------|
| **存量复活引擎** | 激活睡在Excel的200个"前女友" | 批量导入→自动分层（A/B/C）→自动提醒（到期该联系谁） | 20%激活率 = 40个客户，零获客成本 |
| **精准匹配引擎** | 把"能不能贷"变成可解释的缺口+改造方案 | 诊断→产品匹配→缺口识别→改造方案（任务清单+推荐服务商） | 匹配准确率>85%，减少返工 |
| **长期陪跑引擎** | 客户终身价值最大化 | 所有客户都给改造方案（成功也给、失败也给），持续陪跑 | 中介收入提升2.7倍 |

**核心创新点**：
1. **存量复活**：把睡在Excel的客户变成可管理的资产（市场上少有）
2. **方案任务化**：不是给"建议报告"，而是给"可执行任务清单+证据+推荐服务商"
3. **认证服务商网络**：解决"不知道去哪找、谁靠谱"的痛点
4. **白标模式**：每个中介独立产品，数据归自己，不担心被平台抢客户

### 3.3 与竞品的核心差异

| 维度 | 传统助贷CRM | 我们 |
|------|-----------|------|
| **核心定位** | 线索管理+进件效率 | 存量复活+改造执行 |
| **主要场景** | 新客户获取 | 历史客户激活 |
| **核心功能** | 跟单管理 | 批量导入+自动分层+自动提醒 |
| **改造服务** | 无 | 认证服务商网络 |
| **产品形态** | 标准SaaS | 白标定制 |
| **数据归属** | 平台 | 中介公司 |

**我们更聚焦**：
- 聚焦"存量客户复活"而非"新客户效率"
- 聚焦"改造方案执行"而非"线索管理"
- 聚焦"可验收的任务清单"而非"建议报告"

## 四、服务逻辑

### 4.1 统一服务逻辑：所有客户都给改造方案

| 客户类型 | 当前状态 | 给什么 | 为什么 |
|---------|---------|--------|--------|
| **A类-新客已成功** | 能贷80万 | 改造方案（3月后贷150万） | 长期陪跑、客户终身价值最大化 |
| **B类-新客失败** | 贷不了 | 改造方案（3月后能贷） | 失败→成功，变废为宝 |
| **C类-老客失败过** | 睡在Excel | 改造方案（条件够了/改造后能贷） | 复活存量，零获客成本 |
| **D类-老客成功过** | 贷过50万 | 改造方案（下次贷100万） | 增量挖掘，持续创造价值 |

**核心理念**：
> 不管能不能贷、贷多少，都给改造方案。
> 
> 因为"改造"是长期陪跑的核心，是持续创造价值的抓手。
> 
> 把客户牢牢掌握在手里，做企业主的长期顾问，而不是做一锤子买卖。

## 五、核心能力架构（4库+2线）⭐

### 5.1 四个数据库（完整资产）

基座的价值不在于有多少客户，而在于**4个库是否完整、2条线是否跑通**。

| 数据库 | 作用 | 关键字段 | 当前状态 |
|--------|------|---------|----------|
| **客户档案库** | 字段结构化 + 历史申请/失败原因/材料版本 | 客户ID、企业信息、诊断历史、申请记录、失败原因 | ✅ 已设计 |
| **产品库** | 准入条件、额度算法、材料清单、禁入行业、口径说明 | 产品ID、准入条件、额度算法、利率、材料清单 | ✅ 已建设（20个产品） |
| **方案库** | 任务清单+证据要求（不是一段话） | 方案ID、适用场景、任务清单、证据要求、推荐服务商 | ✅ 已建设（12个方案） |
| **结果库** | 成功/失败回填（用来训练"更准"） | 申请ID、客户ID、产品ID、结果、拒绝原因、预测差异 | ⚠️ 待建设（V0必须有） |

**关键洞察**：
> 贷款行业真正烧钱的地方不是"建议"，而是**材料返工、过程失控、经验无法沉淀**。
> 
> 结果库是"越用越准"的核心 - 没有它，经验就无法沉淀，系统永远不会变聪明。

---

### 5.2 两条神经系统（闭环核心）

#### A) 执行线（工作流引擎）⭐

**核心功能**：把"改造建议"变成"可推进的任务"

```
缺口识别 → 任务生成 → 证据要求 → 到期提醒 → 材料齐套检查 → 复评触发
```

**具体实现**（V0最小颗粒度）：

| 功能 | V0要求 | 示例 |
|------|--------|------|
| **任务生成** | 每个缺口至少生成1个任务 | 缺口"开票不足" → 任务"开票提升" |
| **证据要求** | 每个任务明确证据 | 6个月开票记录、增值税申报表 |
| **到期提醒** | 每个任务设置提醒 | 到期前7天提醒 |
| **复评触发** | 30/60/90天自动复评 | 3个月后重新诊断 |
| **版本留痕** | 材料版本管理 | 第1版材料、第2版材料对比 |

**为什么重要**：
- 没有执行线 → 方案永远停留在"建议"层，无法推进
- 有了执行线 → 中介知道"下一步做什么"，客户知道"需要准备什么"

---

#### B) 复盘线（学习闭环）⭐

**核心功能**：让系统越用越准

```
申请结果 → 结果库录入 → 差异分析 → 原因标签化 → 更新规则库 → 匹配准确率提升
```

**具体流程**：

| 步骤 | 做什么 | 谁负责 | 产出 |
|------|-------|--------|------|
| 1. 结果录入 | 每次申请（通过/拒绝）必须录入 | 中介 | 结果库+1条记录 |
| 2. 差异分析 | 对比预测 vs 实际 | 系统自动 | 差异报告 |
| 3. 原因标签 | 拒绝原因归类到6个体质维度 | 系统+人工 | 失败原因库 |
| 4. 规则更新 | 产品口径调整、方案模板优化 | 产品团队 | 规则库更新 |
| 5. 效果验证 | 下次匹配准确率是否提升 | 系统自动 | 准确率报表 |

**数据飞轮**：
```
更多数据 → 更准的匹配 → 更高的通过率 → 更多中介使用 → 更多数据
```

**为什么重要**：
- 没有复盘线 → 失败了也不知道为什么，同样的错误会重复犯
- 有了复盘线 → 系统会自我学习，越用越准，形成数据护城河

---

### 5.3 V0最小闭环（必须能跑通）

**V0必须包含的闭环**：

```
1. 客户档案导入（Excel批量导入）
   ↓
2. 产品库 + 规则匹配（先规则化，别一上来AI）
   ↓
3. 缺口清单（差什么、差多少、影响哪些产品）⭐ 可解释
   ↓
4. 方案输出 = 任务清单 + 证据要求 + 到期提醒 ⭐ 证据化
   ↓
5. 复评（30/60/90天一键重算）
   ↓
6. 结果回填 → 结果库 ⭐ 学习闭环
   ↓
7. 信任底座（租户隔离、操作留痕、审计日志）⭐ 合规
```

**V0产品定义升级**：

```
从：录入 → 匹配 → 报告
到：录入 → 匹配 → 缺口（可解释）→ 任务化（证据化+提醒）→ 复评 → 结果回填
```

**关键判断**：
> **如果V0没有"任务+证据+提醒"，就只是"报告生成器"，不是"基座"。**
> 
> 报告生成器 = 看着专业，但不产生复利，中介不会持续付费。
> 
> 闭环基座 = 能跑起来的系统，越用越准，中介离不开。

| 6个月 | 30家 | 1038万 | 初步验证 |
| 12个月 | 100家 | 3460万 | 区域覆盖 |
| 24个月 | 300家 | 1.04亿 | 全国扩张 |

### 6.2 服务商返佣逻辑（关键）

**情况1：我们推荐的服务商**（核心收入）
```
客户改造费8万 → 服务商返佣1.6万（20%）
→ 我们50%（0.8万）+ 中介50%（0.8万）
```

**前提条件**：
- ✅ 对中介披露："本推荐服务商可能产生返佣分配；返佣比例：服务商返佣的50%；您可自选服务商"
- ✅ 对企业主披露："您可自选服务商，也可使用推荐服务商；推荐服务商已经过认证"
- ✅ 系统内强制勾选（不勾选无法推荐）

**情况2：中介自己找的服务商**
```
客户改造费8万 → 服务商可能给中介返佣
→ 我们不管、不收（100%归中介）
```

**为什么这样设计？**
- 我们推荐的服务商，我们提供了价值（筛选认证、智能推荐、质量监督）→ 分佣
- 中介自己找的，我们没参与 → 不收
- 中介有选择权：用我们推荐的（省心）或自己找（全拿）

### 6.3 披露与对账机制（硬规则）

**披露机制**（写进系统，强制勾选）：
- 对中介：明确告知返佣比例
- 对企业主：明确告知可自选服务商
- 不披露 = 不推荐

**对账机制**：
- 订单号、金额、服务完成定义
- 完成定义：服务商提交验收报告+中介确认
- 退款/争议处理机制
- 系统内可追溯

## 七、第一阶段目标（0-12月）

### 7.1 三条生死线（第一年北极星）⭐

**中介侧**：
- 复活转化率/筛选效率是否能肉眼可见提升
- **否则不续费**

**会计侧**（为Phase 2铺路）：
- 把陪跑变成"每月必做的动作"
- **否则只是一次性咨询**

**合规侧**：
- 不踩雷
- 银行和监管对贷款中介乱象很敏感
- **必须更"正规军"，而不是更像灰产工具**

### 7.2 可验收指标（具体）⭐

**指标1：复活转化率**
- **定义**：导入的存量客户中，被系统触发并推进到有效沟通/进件的比例
- **目标**：>20%
- **验收方式**：系统记录进件数/导入总数

**指标2：筛选周期**
- **定义**：从接触客户到得出"能做/值得做/不建议"的平均时间
- **目标**：<3天
- **验收方式**：系统记录时间戳

**指标3：匹配准确率**
- **定义**：系统推荐的产品中，客户实际申请并通过的比例
- **目标**：>85%（预测）→ 实测
- **验收方式**：结果库回填

**指标4：中介续费率**
- **定义**：第一批中介在12个月后的续费比例
- **目标**：>70%
- **验收方式**：合同续签数据

### 7.3 聚焦两件事

**目标1：产品精准度**（练内功）

系统指标：
- 返工率下降（客户重新申请的次数减少）
- 周期缩短（从诊断到成交的时间缩短）
- 复活转化率（存量客户激活的比例）

业务指标：
- 产品匹配准确率 > 85%
- 改造方案成功率 > 75%（通过案例积累验证）
- 积累100+成功案例
- 积累失败原因库（500+条）

**目标2：中介数量**（验证模式）
- 服务30-100个中介公司
- 验证商业模式可行性
- 建立口碑，形成转介绍

### 7.4 不着急做的（后续迭代）

- ❌ 复杂的服务商认证体系（V1再做）
- ❌ 第二阶段规划（先做好第一阶段）
- ❌ 收入模型优化（先验证模式）
- ❌ 前台获客差异化（第一阶段重点是后台闭环）

**第一阶段策略**：
> 主力投入后台闭环（批量导入→诊断→方案→提醒→复评），
> 门面做标准前台入口（问卷，仅作导流+品牌），
> 不押宝前台获客差异化。

## 八、阶段演进战略（同一系统，三种渠道）⭐

### 8.1 核心原则

**同一个系统、三种分发渠道**：
- 避免"渠道自相残杀"
- 能力内核不变，只是渠道切换
- 每个阶段都是独立的商业模式

**能力内核**（永远不变）：
```
建档（数据）→ 诊断（差距）→ 匹配（当下可做）→ 方案（任务+证据）→ 提醒（时间）→ 复评（闭环）
```

**分发渠道**（三阶段切换）：
```
Phase 1: 贷款中介（最快验证）
Phase 2: 代理记账/会计（陪跑默认化）
Phase 3: 企业直达（最终形态）
```

---

### 8.2 Phase 1：贷款中介（0-12月）⭐ 当前阶段

**为什么先做**：
- 中介手上有"200个前女友"（历史未成交客户）
- 一接就能跑闭环、出成交、拿案例
- 验证最快、反馈最直接

**你卖的不是"获客"，而是**：
- **复活转化率**（导入的存量里，有多少被激活）
- **筛选效率**（从接触到判断"能做/不能做"的时间）
- **方案标准化**（新人也能按清单做）

**成功标准**：
- 30家中介深度使用
- 复活转化率>20%
- 筛选周期<3天
- 中介愿意续费（续费率>70%）

**关键指标**：
```
核心指标：复活转化率、筛选周期
次要指标：匹配准确率、续费率
```

---

### 8.3 Phase 2：代理记账/会计（12-24月）

**为什么上移**：

**优势1：更上游的数据源**
- 票税、流水、报表更真实、更连续
- 会计掌握企业最核心的经营数据

**优势2：更稳定的触点**
- 会计每月都在服务企业
- 陪跑天然发生，不是一次性咨询

**优势3：天然适合"体质改造"**
- 很多改造动作（票税、合规、报表质量、资质）就在他们工作半径里
- 会计推荐的服务商，企业主更容易信任

**包装成**：
- "融资体质仪表盘"
- "行动清单（本月该做什么）"
- "进件准备助手"

**中介角色变化**：
```
Phase 1: 主角（中介主导）
Phase 2: 可选执行方/成交通道（会计主导）
```

**关键提醒（合规）**：
- 数据合规要求更高
- 委托处理/授权留档/可导出删除/审计日志必须是底座能力
- 符合《个人信息保护法》对"委托处理"的明确要求

**成功标准**：
- 100家代理记账公司使用
- 把陪跑变成"每月必做的动作"
- 会计愿意为此付费（作为增值服务）
- 企业主认可度>80%

---

### 8.4 Phase 3：企业直达（24月+）

**企业自己做**：
- 体质管理
- 看"差距—任务—进度—复评"
- 自主选择执行方

**系统提供**：
- 可选的执行网络：银行产品、服务商、合规中介
- 不做强制推荐，企业自主选择

**最大难点**：
- 不是产品，是获客与信任
- 前两阶段的口碑与数据闭环很关键
- 需要品牌建设和市场教育

**成功标准**：
- 直接服务1000+企业
- 企业自主使用率>50%
- NPS>40
- 形成品牌认知度

---

### 8.5 阶段切换触发条件

**Phase 1 → Phase 2**：
- ✅ 30家中介稳定续费
- ✅ 案例库>100个
- ✅ 复活转化率>20%
- ✅ 有5家会计公司主动要求接入

**Phase 2 → Phase 3**：
- ✅ 100家会计公司使用
- ✅ 企业端有明确需求（>50家企业主动询问）
- ✅ 品牌认知度足够（行业内知名度>30%）
- ✅ 系统成熟度足够（匹配准确率>90%）

**关键判断**：
> 不要着急进入下一阶段。
> 
> 每个阶段都要"跑透"——不仅是数字达标，更重要的是模式验证成功、口碑建立、团队能力具备。
> 
> 过早进入下一阶段，会导致"两头不到岸"。

---

### 8.6 护城河演进

**Phase 1护城河**：
- **存量复活闭环**（市场少有）
- **方案任务化**（比报告更实用）
- **4库+2线**（结构化资产）

**Phase 2护城河**：
- **数据闭环**（会计数据更真实）
- **陪跑默认化**（每月必做）
- **结果库积累**（越用越准）

**Phase 3护城河**：
- **结果库+复盘线**（数据飞轮）
- **品牌信任**（口碑积累）
- **网络效应**（服务商网络、银行合作）

**护城河演进逻辑**：
```
Phase 1: 产品能力（闭环跑通）
Phase 2: 数据能力（越用越准）
Phase 3: 生态能力（网络效应）
```

---

### 8.7 为什么这样设计（战略逻辑）

**传统错误做法**：
```
❌ 一开始就做企业端平台
→ 获客成本高、信任难建立、教育成本高
→ 很多创业公司死在这里

❌ 同时做中介和企业
→ 渠道冲突、定位不清、资源分散
→ 两头不到岸
```

**我们的做法**：
```
✅ 先用强场景把系统跑起来（中介的存量复活）
✅ 然后把分发渠道换成更上游、更稳定的节点（会计）
✅ 最后才直达企业（此时已有口碑和数据）
```

**关键洞察**：
> 渠道可以换，能力不能换。
> 
> 我们始终卖的是同一个能力：把融资从"碰运气"变成"可解释、可执行、可复评"。
> 
> 只是在不同阶段，通过不同渠道，服务不同角色。

## 九、三大硬规则

### 9.1 白标配置边界（避免外包地狱）

**可配置（标准服务，所有中介都能自己设置）**：
```
✅ Logo、主色、域名
✅ 公司名称、联系方式
✅ 产品库启停（勾选展示哪些产品）
✅ 方案库启停（勾选展示哪些方案）
✅ 自定义字段（≤5个）
✅ 话术模板（可编辑）
```

**不定制（硬规则，超出即产品化）**：
```
❌ 任何流程/算法改动
→ 只能进"公共产品迭代池"
→ 不做单客户专属改造
```

**为什么？**
> 白标SaaS = 同一套基座、不同配置
> 
> 不是 = 同一套人力、不同定制
> 
> 如果超出配置边界，要么拒绝，要么产品化（所有客户都能用）。

### 9.2 数据权属机制（信任底座）

**数据权属**：
- 中介公司：数据控制方/处理者
- 我们：受托处理方

**三个可验收机制**：

**机制1：租户隔离+操作留痕**
- 不同中介的数据完全隔离（物理或逻辑）
- 所有操作可审计（谁导入/查看/导出/删除，全记录）
- 审计日志至少保留1年

**机制2：随时可导出/删除**
- 中介随时可导出所有数据（Excel/JSON）
- 终止合作可要求删除所有数据
- 7个工作日内完成删除并提供证明

**机制3：最小化采集+授权留档**
- 只收集13个必要字段
- 批量导入时要有"客户授权证明"
- 每个字段都说清楚用来干什么

**委托处理协议**（V0必须有）：
- 明确双方权利义务
- 明确数据使用范围
- 明确违约责任

### 9.3 合规与信任（P0级）

**合规表述规范**：

禁止表述：
- ❌ "低息快贷"
- ❌ "无条件放款"
- ❌ "百分百放款"
- ❌ "内部渠道"

统一措辞：
- ✅ "初步评估/预估/建议"
- ✅ "以银行审批为准"
- ✅ "仅供参考"

**算法护栏**：

护栏1：可解释
- 每个分层都能说清楚为什么（基于哪些字段、哪些规则）
- 每个缺口都能量化（差多少、影响什么）

护栏2：人工复核
- 中介可以改判系统的分层（A→B或B→A）
- 改判时要记录理由
- 系统学习人工修正

护栏3：免责声明
- 固化在系统里（每个评估报告都带）
- 明确"不承诺通过/额度/利率"
- 明确"以银行审批为准"

## 十一、三个边界确认（钉死）

### 边界1：改造方案 = 任务清单+推荐服务商

**确认**：✅ 是的

**明确定义**：
```
我们交付的"改造方案"包含：
1. 缺口分析（差什么、差多少）
2. 任务清单（做什么、谁负责、证据要求、完成标准）
3. 推荐服务商（2-3家可选，或中介自己找）
4. 效果预览（改造后预计能贷多少）⚠️ 以银行审批为准
5. 成本估算（改造成本、ROI）

我们不交付：
❌ 改造执行（不做H业务/高新认定等）
❌ 承担改造风险（服务商负责）
```

### 边界2：数据权属 = 中介是控制方

**确认**：✅ 是的

**明确定义**：
```
数据权属：
- 中介公司：数据控制方/处理者
- 我们：受托处理方

中介的权利：
✅ 随时导出所有数据（Excel/JSON）
✅ 终止合作可要求删除
✅ 数据隔离（其他中介看不到）

我们的义务：
✅ 最小化采集（13个必要字段）
✅ 授权留档（导入时要有授权证明）
✅ 操作留痕（审计日志）
✅ 租户隔离（物理或逻辑）
```

### 边界3：白标定制 = 只做配置

**确认**：✅ 是的

**明确边界**：
```
可配置（标准服务）：
✅ Logo、颜色、域名
✅ 公司名称、联系方式
✅ 产品库/方案库启停
✅ 自定义字段（≤5个）
✅ 话术模板

不定制（硬规则）：
❌ 流程级改动（改诊断逻辑、匹配算法）
❌ 第三方接口对接（除非所有客户都需要）
❌ 个性化开发（除非升级为全局功能）

超出配置边界 → 要么拒绝，要么产品化（所有客户都能用）
```

### 边界4：双语词典（对外中性，对内接地气）⭐

**确认**：✅ 是的

**原则**：
- 对外（企业端、会计端、合规）：使用中性词汇
- 对内（中介侧、团队内部）：使用接地气词汇
- 不同场景自动切换，避免误解

**词汇对照表**：

| 场景 | 对外表述 | 对内表述 |
|------|---------|---------|
| 企业端/会计端 | 提升路径 | 改造方案 |
| 企业端/会计端 | 进件准备清单 | 材料清单 |
| 合规/银行 | 融资体质评估 | 诊断 |
| 正式文档 | 历史未成交客户 | 沉睡客户/前女友 |
| 企业端/会计端 | 融资体质仪表盘 | 诊断系统 |
| 合规/银行 | 初步评估/建议 | 诊断结果 |

**为什么需要双语词典？**

**对中介（接地气）**：
- "改造"一听就懂，很直接
- "前女友"很形象，立即get到存量复活的意思
- "诊断"比"评估"更有专业感

**对企业/会计（中性专业）**：
- "改造"容易被误解为"我们要交付工程"
- "前女友"不够正式，可能显得不专业
- "提升路径"听起来更中性、更可接受

**对合规/银行（严谨中立）**：
- "诊断"要说成"初步评估"（避免医疗类比）
- "方案"要说成"建议"（避免承诺性表述）
- "改造"要说成"提升路径"（更中性）

**系统实现**：
- 系统内部有"语言模式"设置
- 根据不同租户类型（中介/会计/企业），自动切换措辞
- 文档生成时，根据目标受众选择不同词汇

**例子**：

```markdown
# 中介版报告
诊断结果：您的客户A001需要改造
缺口：开票不足（差20万）
推荐方案：开票提升术（改造方案002）
```

## 附录A：核心话术（长期陪跑）

### A类客户话术（已经能贷，为什么还要改造）

```
"王总，恭喜您，现在条件符合，能贷到80万。

我给您两个方案：

1. 现在立即申请80万（着急用钱）
2. 做个改造，3个月后申请150万（多拿70万）

我的建议：
如果不是特别着急，建议等3个月。
因为做了改造，不仅这次能多贷70万，
以后您的企业条件更好，贷款会越来越容易。

我是您的长期顾问，不是做一锤子买卖。
我希望帮您把企业体质养好，
以后需要钱，随时能贷到。

要不要试试第2个方案？"
```

**核心逻辑**：
- 给选择（不强迫）
- 讲长期价值（体质好 → 持续能贷）
- 定位长期顾问（不是一次性服务）

## 附录B：验证计划

### B.1 如何验证基座可用？

**验证方式**：
1. 找3个中介深度测试
2. 每个中介导入5-10个真实历史客户
3. 让中介全流程操作一遍：
   - 批量导入客户
   - 查看诊断报告
   - 选择改造方案
   - 跟进A类客户
   - 设置B类客户提醒
4. 收集反馈

**关键验证点**：
- 批量导入是否真的有用？
- 自动分层（A/B/C）是否准确？
- 改造方案是否真的可执行？
- 自动提醒是否真的有价值？

### B.2 成功标准

**最低标准**：
- 3个中介都说"批量导入有用"
- 至少2个愿意导入真实历史客户
- 至少1个愿意付费试用（哪怕是199元/月）

**理想标准**：
- 3个中介都愿意导入历史客户
- 3个中介都愿意付费
- 至少1个愿意签年费合同
- 至少1个成功激活了存量客户

**失败退出机制**：
- 如果3个中介都说"没用" → Kill项目
- 如果2个说"有用但不付费" → Pivot定价或功能
- 如果1个说"有用愿意付费" → 继续迭代
