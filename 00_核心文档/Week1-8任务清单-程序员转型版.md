## 🎯 这个文档是干什么的

8周培训的具体任务清单和验收标准。

**你在这8周要完成的事**：把5个真实银行产品翻译成Skill配置，写代码让它们跑通，做一个前端让中介能实际用起来，最后部署上线。

**核心原则**：Skill设计和验证是主线任务（占60%时间），写代码是让Skill跑起来的手段（占40%时间）。

**使用方式**：
- 候选人：只看Week 1-2
- 已入职：按周执行，每周末对照验收Checklist自查

## Week 1：环境搭建 + 理解贷款业务 + 第一个Skill（Day 1-5）

### 目标

搭好开发环境，理解"贷查查是干什么的"，写出你的第一个Skill配置文件。

Week 1结束时你应该能：
- 用自己的话讲清楚"贷查查解决什么问题"
- 看懂一个Skill文件，知道每个字段什么意思
- 手写一个简单的Skill（照着模板改参数即可）

### 任务1.1：开发环境搭建（Day 1）

- [ ] Python 3.11+ 安装成功
- [ ] Node.js 18+ 安装成功
- [ ] Git配置完成，项目代码拉取成功，能运行
- **验收**：pytest和npm build都能通过

### 任务1.2：产品理解（Day 2-3）

- [ ] 阅读总纲v1.1（路径：`00_核心文档/贷款中介业务基座-总纲-v1.1.md`，理解项目定位）
- [ ] 阅读银行政策库中任意3个产品（路径：`01_银行政策库/`，下面有税贷/、兴业银行-兴业管家贷.md、平安银行-小微企业信用贷.md等，任选3个阅读）
- [ ] 写产品理解文档：用自己的话写清楚"贷查查是干什么的""目标用户是谁""核心价值是什么"
- [ ] 提出至少1个问题（为什么这么做？有没有更好的方式？）
- **验收**：能对着创始人讲5分钟，讲清楚产品是干什么的

### 任务1.3：第一个Skill配置（Day 4-5）

**背景**：Skill就是一个YAML文件，把银行产品的准入条件、额度计算公式、报告模板写成机器能读的格式。

**任务**：
- [ ] 学习YAML语法（10分钟教程即可）
- [ ] 阅读《Skill格式规范》（路径：`05_开发文档/Skill格式规范-v1.0.md`）
- [ ] 从银行政策库中选1个产品，翻译成Skill配置文件
- [ ] Skill必须包含：准入条件（至少3个）+ 额度计算公式 + 报告模板
- [ ] 所有金额字段明确标注单位
- **验收**：Skill通过jsonschema验证；能解释每个字段的含义（不是抄模板）

### 任务1.4：竞品分析（Day 5，与1.3并行）

- [ ] 找到至少2个竞品（贷款中介SaaS工具）
- [ ] 对比功能差异
- [ ] 分析"我们和竞品有什么区别"
- **验收**：能说出至少1个我们的差异化优势

### Week 1验收标准（二值判断）

- [ ] 环境能跑通
- [ ] 产品理解文档已提交，能讲清楚
- [ ] 第一个Skill文件已提交，jsonschema验证通过
- [ ] 竞品分析已提交

### 如果Week 1通不过

- 再给1周补课
- Week 2还不通过 → 友好结束，给书面反馈

**Day 6-7**：周末/缓冲日（未安排任务，可用于补课或休息）

## Week 2：Skill系统深入 + 匹配程序雏形（Day 8-10）

### 目标

多写几个Skill，理解不同银行产品的规则差异。开始写"让Skill跑起来"的匹配程序。

Week 2结束时你应该能：
- 独立翻译一个新的银行产品为Skill（不看模板）
- 能说出"不同产品的Skill之间，结构相同但参数不同"
- 写出匹配程序的最小版（输入客户数据 + 1个Skill → 输出符合/不符合 + 原因）

### 任务2.1：再写2个Skill（Day 8-9）

- [ ] 从银行政策库中再选2个产品，翻译成Skill
- [ ] 注意：不同产品的准入条件差异大（有的看流水、有的看纳税、有的看资质）
- [ ] 每个Skill写1-2个测试数据（"这个客户应该符合/不符合"）
- **验收**：3个Skill（含Week 1的1个）都通过jsonschema验证；能解释3个产品的准入条件有什么不同

### 任务2.2：匹配程序最小版（Day 9-10）

**什么是匹配程序**：一段代码，做的事情是：拿客户数据 → 读Skill文件 → 逐条检查准入条件 → 输出"符合/不符合 + 原因 + 预估额度"。

**现有框架**：项目里已有外壳代码，先看明白再动手：`05_开发文档/ProductMatcherRunner原型代码-v1.0.md`（包含目录结构、输入输出格式、核心Runner类）。你的任务是在现有结构里填充匹配逻辑，不是从零设计。如果对框架有疑问，找导师对一遍。

**任务**：
- [ ] 先阅读ProductMatcherRunner原型代码，理解输入输出格式和目录结构
- [ ] 写一个Python函数：输入=客户数据(dict) + Skill文件路径 → 输出=匹配结果
- [ ] 支持最基本的操作符：>=、<=、==、in
- [ ] 支持简单的额度计算（如annual_tax * 25）
- [ ] 用AI辅助生成代码（Cursor/Claude），但你必须看懂每一行，能解释逻辑
- [ ] 用Week 2.1的测试数据验证：结果是否符合预期
- **验收**：3个Skill都能正确匹配；测试数据全部通过；能说清楚"这段代码做了什么"

### 任务2.3：产品思维验收（Day 10）

- [ ] 能回答"三拷问"：输入是什么/处理是什么/输出是什么

  参考答案（对着匹配程序）：输入=客户的13个字段 + 所有Skill文件；处理=逐条比对准入条件、计算额度公式、生成解释文本；输出=能贷的产品列表（含预估额度）+ 不能贷的产品列表（含差距原因）。不需要背过来，用自己的话说清楚即可。

- [ ] 能说出"为什么用Skill配置而不是硬编码if-else"
- [ ] 能提出至少1个优化建议（关于Skill结构、匹配逻辑、或报告模板）

### Week 2验收标准（二值判断）

- [ ] 3个Skill文件已提交，验证通过
- [ ] 匹配程序能运行，3个Skill都能正确匹配
- [ ] 测试数据全部通过
- [ ] 能回答"三拷问"

## Week 3-4：覆盖5个产品 + 匹配程序完善 + 质量管控（Day 11-18）

### 目标

把银行政策库中的5个产品全部翻译成Skill，同时让匹配程序能处理更复杂的情况。建立代码质量习惯。

这两周结束时你应该能：
- 独立管理5个Skill文件，任何一个改了都能快速验证
- 匹配程序能批量处理（输入1个客户 → 扫描所有Skill → 输出"能贷哪些、不能贷哪些"）
- 代码有测试、有格式检查、有版本记录

### 任务3.1：翻译剩余2个产品为Skill（Day 11-12）

**5个产品的选取原则**：5个产品尽量覆盖不同的准入条件类型，建议组合如下（最终选哪5个找导师确认）：
- 税贷类（看开票额/纳税额，如农业银行纳税e贷、民生税贷、浦发浦慧税贷）
- 流水类（看对公流水，如兴业管家贷）
- 信用类（看征信报告，如平安小微企业信用贷）

不需要自己决定，Week 1末和导师对一遍确认名单即可。

- [ ] 从银行政策库中再选2个产品（尽量选条件类型不同的，比如一个看资质、一个看资产）
- [ ] 翻译成Skill配置文件
- [ ] 每个Skill写测试数据（至少2组：1组符合、1组不符合）
- **验收**：5个Skill全部通过jsonschema验证

### 任务3.2：匹配程序升级——批量扫描（Day 12-14）

**需求**：之前的程序是"1个客户 vs 1个Skill"。现在要升级为"1个客户 vs 所有Skill"，输出一份完整的匹配报告。

- [ ] 程序能读取一个目录下的所有Skill文件
- [ ] 输入1个客户数据 → 逐个匹配 → 输出：符合的产品列表（含预估额度）+ 不符合的产品列表（含不符合原因）
- [ ] 支持更多操作符（如有需要，根据实际Skill中的条件扩展）
- [ ] 用AI辅助生成代码，但每个函数必须有注释说明"这段做什么"
- **验收**：输入1个测试客户 → 输出5个产品的匹配结果 → 结果与人工判断一致

### 任务3.3：Skill热更新（Day 15-16）

**需求**：改了Skill文件后，程序能重新加载，不用重启。

- [ ] 写一个API：POST /api/reload-skills → 重新读取所有Skill文件 → 返回加载报告（成功几个、失败几个）
- [ ] 如果某个Skill文件格式错误，不影响其他Skill（错误的跳过，正确的继续用）
- [ ] 如果全部加载失败，保留上一版（不能让系统空了）
- **验收**：改一个Skill文件 → 调用reload → 立即生效；故意写一个格式错误的Skill → reload后其他Skill不受影响

### 任务3.4：质量管控（Day 17-18）

- [ ] 用pytest为匹配逻辑写测试，覆盖率 ≥ 60%
- [ ] 写单位陷阱测试（防止万元/元混用导致1万倍错误）
- [ ] 写边界测试（刚好等于阈值、刚好超过阈值）
- [ ] 用ruff格式检查通过
- [ ] 写README：说明"这个程序做什么、怎么运行、怎么加新Skill"
- **验收**：pytest全部通过；ruff无报错；README能让一个新人看懂

### Week 3-4验收（现场演示，1小时）

**演示内容**：
1. 展示5个Skill文件，讲解"这5个产品的准入条件有什么不同"
2. 输入一个测试客户，现场跑匹配，看结果对不对
3. 现场改一个Skill的条件，reload，再跑一次，看是否生效
4. 展示测试用例，特别是单位陷阱测试

**必答问题**：
- "如果银行调整了某个产品的准入条件，你需要改什么？"（答案：只改Skill文件，不改代码）
- "如果要加一个新的银行产品，步骤是什么？"（答案：写一个新Skill文件，reload即可）
- "万元和元搞混了会怎样？你怎么防？"（答案：单位陷阱测试）

## Week 5-6：前端 + 报告设计（Day 19-26）

### 目标

让中介能实际用起来：填客户信息 → 看匹配报告。重点不是"前端好不好看"，而是"报告中介能不能看懂、能不能用"。

这两周结束时你应该能：
- 中介填完13字段 → 立即看到ABC分层报告
- 报告内容：体质评分 + 能贷哪些产品 + 预估额度 + 不能贷的差在哪 + 改造建议
- 一个不懂贷款的人看完报告能理解"我这个客户什么情况"

### 任务5.1：设计报告模板（Day 19-20）

**这是本阶段最重要的任务**。在写前端代码之前，先想清楚报告长什么样。

- [ ] 定义报告结构：哪些信息必须展示、按什么顺序、怎么分层
- [ ] 设计ABC分层逻辑：什么样的客户是A（直接可贷）、B（改造后可贷）、C（暂不可贷）
- [ ] 写3个示例报告（用markdown或Word）：分别对应A/B/C三类客户
- [ ] 拿给创始人review："中介看到这个报告，能不能直接去跟客户沟通？"
- **验收**：3份示例报告，创始人确认"能用"

### 任务5.2：13字段输入表单（Day 21-23）

- [ ] 用React + Ant Design做表单
- [ ] 13个字段，每个有验证规则、tooltip说明、单位提示
- [ ] 提交到后端API → 调用匹配程序 → 返回结果
- [ ] 处理加载状态、错误提示
- **验收**：能正常提交，异常情况有友好提示

### 任务5.3：报告展示页（Day 23-26）

- [ ] 按5.1设计的模板做前端展示
- [ ] ABC分层用颜色区分（A绿/B黄/C红）
- [ ] 每个匹配产品展示：产品名、预估额度、匹配原因
- [ ] 不匹配的产品展示：差在哪里、需要改造什么
- [ ] 合规固定件：免责声明（"仅供初步评估/以实际审批为准"）
- **验收**：端到端跑通（填表 → 看报告）；找一个非技术人员看报告，能看懂

### Week 5-6验收（现场演示，1小时）

**演示内容**：
1. 现场填一个客户信息，提交，看报告
2. 讲解"报告里每一块信息是怎么来的"
3. 故意填一个极端数据，看异常处理

**必答问题**：
- "中介看到这个报告，下一步会做什么？"（答案：跟客户沟通/开始改造/换产品）
- "如果报告里的预估额度和银行实际审批差很多，怎么办？"（答案：报告已标注"以实际审批为准"，预估只是参考）

## 培养成功的标志

### 8周后，如果这个实习生：

**技术能力**：能独立完成一个功能模块；能写出 60%+ 覆盖率的测试；能用 AI 辅助开发（知道 AI 的局限性）；能写清晰的技术文档。

- **技术OK，产品思维弱** → 纯程序员，继续做开发不参与产品决策

- 通过标准：总分≥70，技术≥28，Skill设计≥20

## 💡 常见问题

**Q1：为什么要写这么多文档？**

- 文档是思考的产物，能写清楚 = 能想清楚

- A：理解"我们和竞品有什么区别"
- 能判断产品定位，提出差异化建议

**Q3：为什么要写测试？**
- A：确保代码质量，避免bug
- 没有测试 = 未交付
