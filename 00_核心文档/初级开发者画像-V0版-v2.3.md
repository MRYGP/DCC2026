### 你们要什么样的人？（5分钟）

### 不是什么

- ❌ 传统实习生（需要手把手教）
- ❌ 纯AI操作员（只会复制粘贴AI代码）
- ❌ 架构师（追求完美设计）
- ❌ 算法竞赛选手（刷题高手但工程能力弱）

### 而是什么

**"能把银行产品翻译成可执行配置、能验证AI输出的产品工程师"**：

**核心能力转变**：
- ❌ 传统程序员：写业务代码
- ✅ 产品工程师：**写Skill配置 + 写代码让Skill跑通 + 验证输出**

**具体体现**：
- ✅ 会写Skill（YAML配置）：把银行产品规则翻译成机器能读的格式
- ✅ 会写匹配程序：让Skill跑起来，输出符合/不符合+原因+额度
- ✅ 会验证AI输出：确保匹配结果符合业务逻辑
- ✅ 会热更新配置：改Skill不用重启
- ✅ 会用AI辅助写代码：但知道AI生成的代码要自己验证

**能力公式**（修正版）：

```
价值 = 业务理解（30%）+ Skill设计能力（40%）+ 代码与验证能力（30%）
```

**关键区别**：

- 传统开发：理解需求 → 写代码 → 测试 → 交付
- **Skill驱动开发**：理解银行产品 → 写Skill配置 → 写代码让Skill跑通 → 验证输出 → 交付
- 区别：代码量更少，但对业务理解的要求更高

**DCC2026的Skill系统就是典型案例**：
- Skill = 你写的"产品规则配置"（YAML）
- 匹配程序 = 让Skill跑起来的那段代码
- 你的价值 = 设计规则，让一段通用程序能处理几十个产品

### DCC2026与通用Web开发的核心区别

**通用Web开发**：
- 主要工作 = 写业务代码
- 验收标准 = 功能能跑

**DCC2026**：
- 主要工作 = 把银行产品翻译成Skill配置 + 写代码让配置跑通
- 验收标准 = Skill能正确匹配 + 报告中介能看懂 + 代码通过质量闸门

**核心差异**：
- 60%时间在理解业务和设计Skill，40%时间在写代码
- 代码不多，但必须可靠（金融场景不容出错）
- 改产品规则 = 改Skill文件，不改代码

### 验收1：Skill系统能力（给出具体案例）

**为什么Skill能力是第一个验收项**：因为Skill设计是实习生的主线工作。写代码的能力在验收2-5中考查，但如果Skill写不好，代码再好也没用——你的代码是为了让Skill跑通而存在的。

**验收任务**：30分钟内完成"工商银行税贷"产品Skill

**要求**：
- [ ] Skill包含：3个准入条件（流水/纳税/征信）
- [ ] Skill包含：1个额度计算公式（如 annual_tax * 25）
- [ ] 所有金额字段明确标注单位（元 or 万元）
- [ ] Skill通过jsonschema验证（无格式错误）
- [ ] 能写至少1个测试用例（输入客户数据 → 预期输出）
- [ ] 能解释每个字段的含义（不能只是抄模板）

**测试输入**：
```python
customer = {
    "monthly_revenue": 120000,  # 12万元（月均流水）
    "annual_tax": 18000,        # 1.8万元（年纳税）
    "credit_query_count": 5     # 征信查询5次
}
```

**预期输出**：
```python
{
    "eligible": True,              # 符合准入
    "amount_estimate": 450000,     # 预估额度45万元（1.8万 × 25）
    "reasons": [
        "流水满足：12万 >= 10万",
        "纳税满足：1.8万 >= 1万",
        "征信满足：5次 <= 6次"
    ]
}
```

**不通过标准**：
- 30分钟内未完成
- Skill格式错误（jsonschema验证失败）
- 单位不明确（无法判断是元还是万元）
- 无法解释字段含义（只是抄模板）

### ⚠️ 字段单位口径决策（V0强制）

> **⚠️ 待决策**：截止日期2026-02-20。请确认最终选择方案A（统一用元）还是方案B（统一用万元），确认后修改所有Skill文件、代码示例和培训材料。

**现状**：
- SSOT定义：13字段单位为「万元」
- 代码示例/培训文档：部分使用「元」（150000元 vs 15万元）
- 风险：混用会导致1万倍计算错误

**V0决策（必须在2026-02-20前确定）**：

**方案A（推荐）**：统一改为「元」
- 理由：代码直观（150000元 vs 15万元），减少小数
- 成本：修改SSOT文档（走变更流程）
- 影响：所有Skill/代码/培训材料统一改为「元」

**方案B（备选）**：统一改为「万元」
- 理由：对齐SSOT，减少变更
- 成本：修改所有代码示例、培训材料
- 影响：代码中需要注意小数精度（15.5万元）

**V0过渡执行规则（在决策前）**：
```python
# 输入层：明确标注单位
class CustomerInputDTO:
    monthly_revenue: float  # 单位：万元（用户输入）
    annual_tax: float       # 单位：万元（用户输入）

# 内部引擎：统一用「元」（临时决定）
class Customer:
    monthly_revenue: float  # 单位：元（引擎内部）
    annual_tax: float       # 单位：元（引擎内部）

# 转换器（V0临时方案）
def normalize_input(dto: CustomerInputDTO) -> Customer:
    """输入层转换：万元→元"""
    return Customer(
        monthly_revenue=dto.monthly_revenue * 10000,  # 万元→元
        annual_tax=dto.annual_tax * 10000
    )
```

**强制测试用例（V0必须）**：
```python
def test_unit_trap_10000x_error():
    """单位陷阱测试：防止1万倍错误"""
    # 输入：10万元（用户以「万元」输入）
    dto = CustomerInputDTO(monthly_revenue=10)  # 10万元

    # 转换后：100000元
    customer = normalize_input(dto)
    assert customer.monthly_revenue == 100000  # 元

    # Skill阈值：100000元
    skill = load_skill("PRODUCT_XXX")
    assert skill.eligibility.conditions[0].value == 100000  # 元

    # 判断结果：应该通过（100000 >= 100000）
    assert check_eligibility(customer, skill) == True

def test_unit_consistency_across_skills():
    """单位一致性测试：所有Skill阈值单位统一"""
    skills = load_all_skills()
    for skill in skills:
        for condition in skill.eligibility.conditions:
            if condition.field in ["monthly_revenue", "annual_tax"]:
                # 所有金额类阈值必须 >= 1000（说明用的是「元」而非「万元」）

```

**Poetry**（依赖管理）+ **ruff**（lint+format）+ **pre-commit**（推荐）。

### 第3层：AI工具熟练度（核心竞争力）

#### 3.1 AI编程助手（必须精通至少一个）

**V0推荐组合**：主力 Cursor + 辅助 Claude Code。

**关键能力**：会用自然语言描述需求；**会验证AI生成的代码**；**会让AI代码通过质量闸门**；会在AI出错时调试。

#### 3.2 质量闸门（V0强制）

- `ruff format --check .` / `ruff check .`
- `pytest tests/ --cov --cov-report=term-missing`
- `bandit -r app/ -ll`

**闸门规则**：pytest覆盖率 ≥ 60%；格式检查通过；无明显安全漏洞。**不通过闸门的代码 = 未交付**

#### 3.3 前端最小闸门（V0必须）

- `npm run build` 必须通过
- `npm run lint` 错误不能有
- `npm audit --production`（V0可选）

#### 3.4 安全升级意识（V0硬性要求）

会看安全公告、识别受影响版本、升级依赖、跑回归测试、验证修复。至少处理过1次依赖升级。
