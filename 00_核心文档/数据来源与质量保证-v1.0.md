## 一、数据架构全景

### 1.1 4类数据库总览
```
产品运行 = 4个数据库协同工作

├─ 1. 客户档案库（中介输入）⭐ 核心输入
│   ├─ 数据来源：中介手工输入（Excel/H5表单）
│   ├─ 数据质量：中等（使用区间而非精确值）
│   ├─ 获取难度：⭐⭐ 中等
│   ├─ 维护成本：0元（中介负责）
│   └─ 关键风险：中介懒得填、客户不愿透露真实数据

├─ 2. 产品库（我们维护）⭐ 知识资产
│   ├─ 数据来源：银行官网+客户经理+中介反馈
│   ├─ 数据质量：高（多渠道核验）
│   ├─ 获取难度：⭐⭐⭐ 难
│   ├─ 维护成本：10-15小时/月（V0），2小时/月（V2）
│   └─ 关键风险：政策变化未及时发现

├─ 3. 方案库（我们维护）⭐ 知识资产
│   ├─ 数据来源：服务商报价+案例积累
│   ├─ 数据质量：中（基于案例统计）
│   ├─ 获取难度：⭐⭐ 中等
│   ├─ 维护成本：2小时/月
│   └─ 关键风险：成功率估计不准确

└─ 4. 结果库（中介回填）⭐ 数据飞轮
    ├─ 数据来源：中介回填申请结果
    ├─ 数据质量：低→高（取决于回填率）
    ├─ 获取难度：⭐⭐⭐⭐ 最难
    ├─ 维护成本：0元（中介负责）
    └─ 关键风险：回填率<50%，数据飞轮无法启动
```

### 1.2 数据流转关系
```
输入层（客户档案库）
  ├─ 中介导入客户数据（13字段）
  └─ 系统存储到customers表

    ↓

知识层（产品库 + 方案库）
  ├─ 产品库：20个银行产品（我们维护）
  ├─ 方案库：12个改造方案（我们维护）
  └─ 诊断逻辑：六宫格评分算法

    ↓

处理层（诊断引擎）
  ├─ 客户数据 + 产品库 → 匹配算法
  ├─ 客户数据 + 方案库 → 差距诊断
  └─ 输出：ABC分层报告 + 改造建议

    ↓

反馈层（结果库）
  ├─ 中介回填申请结果（通过/拒绝）
  ├─ 系统记录到result_feedback表
  └─ 数据飞轮：结果 → 规则优化 → 更准确的推荐

    ↓

优化层（持续迭代）
  ├─ 分析结果库：发现规律
  ├─ 调整诊断规则：优化阈值
  ├─ 更新产品库：标记不准确的产品
  └─ 准确率提升：85% → 92% → 95%
```

## 二、客户档案库（13字段）

### 2.1 数据来源方式

**方式A：Excel批量导入**（成熟期中介，200+客户）
```yaml
流程：
  1. 中介下载Excel模板（13列）
  2. 中介填写客户数据
  3. 中介上传Excel文件
  4. 系统解析并写入数据库

适用对象：
  - 成熟期中介（200+客户）
  - 有Excel整理能力
  - 一次性导入大量历史客户

优势：
  ✅ 批量处理（效率高）
  ✅ 一次性完成
  ✅ 历史客户快速迁移

劣势：
  ⚠️ 数据质量可能参差不齐
  ⚠️ 需要中介有Excel能力
  ⚠️ 格式错误需要人工处理

技术实现：
  - 后端：Python pandas库解析Excel
  - 前端：文件上传组件
  - 校验：逐行校验，批量写入
  - 错误处理：标记错误行，提示修改
```

**方式B：H5表单逐个填写**（新手/成长期中介，0-200客户）
```yaml
流程：
  1. 中介打开H5表单
  2. 填写13字段（3-5分钟/个）
  3. 提交 → 写入数据库
  4. 继续添加下一个客户

适用对象：
  - 新手中介（0-50客户）
  - 成长期中介（50-200客户）
  - 逐个录入新客户

优势：
  ✅ 数据质量高（即时填写）
  ✅ 引导性强（帮助文案/下拉选项）
  ✅ 实时校验（避免格式错误）
  ✅ 手机端友好（随时随地填写）

劣势：
  ⚠️ 逐个填写（效率低）
  ⚠️ 不适合大批量（200+客户太慢）

技术实现：
  - 前端：H5响应式表单
  - 实时校验：前端+后端双重校验
  - 进度提示：第X步/共4步
  - 自动保存：避免数据丢失
```

### 2.3 数据质量问题与应对策略

#### 问题1：客户不愿透露真实数据

**风险描述**：
```
- 营收、开票、纳税可能被低报/高报
- 客户担心数据泄露
- 客户担心被税务稽查
```

**应对策略**：
```yaml
策略1：用区间而非精确值
  - 营收："100-300万"而非"235万"
  - 降低心理门槛
  - 减少数据泄露风险

策略2：免责声明
  - "初步评估，以银行实际审批为准"
  - "数据仅用于诊断，不会泄露"
  - 消除客户顾虑

策略3：鼓励真实填写
  - "数据越准确，匹配越精准"
  - "避免浪费时间申请不合适的产品"
  - 强调对客户的好处

策略4：隐私承诺
  - "数据加密存储"
  - "仅用于诊断，不用于其他用途"
  - "中介不会看到原始数据"
  - 建立信任
```

#### 问题2：客户记不清楚

**风险描述**：
```
- 连续开票时长、纳税信用等级等可能记不准
- 房产估值可能高估/低估
- 当前贷款利率可能记错
```

**应对策略**：
```yaml
策略1：给"不清楚"选项
  - 纳税信用等级："A/B/C/M/不清楚"
  - 系统按保守估计（不清楚=C级）
  - 避免强迫客户瞎填

策略2：帮助文案
  - "不知道纳税等级？登录电子税务局查询"
  - "不知道房产估值？参考周边房价"
  - 引导客户找到数据

策略3：给典型区间选项
  - 连续开票时长："<6月/6-12月/12-24月/>24月"
  - 避免填空（减少认知负担）
  - 提高填写完成率

策略4：容错机制
  - 不清楚的字段按保守估计
  - 系统多次推荐都被拒绝 → 提示"请重新核实数据"
  - 允许后续修改完善
```

#### 问题3：中介懒得填

**风险描述**：
```
- Excel模板13列，中介觉得麻烦
- H5表单3-5分钟，中介没时间
- 中介觉得投入产出比不高
```

**应对策略**：
```yaml
策略1：极简字段
  - 只要13列（不能再少）
  - 每个字段都有明确价值
  - 删除任何可有可无的字段

策略2：批量导入
  - Excel一次性解决200个客户
  - 提供模板和填写示例
  - 5分钟导入 vs 10小时逐个填写

策略3：价值承诺
  - "5分钟导入，立即知道谁能贷"
  - "节省80%筛选时间"
  - "提高成交率30%"
  - 强调对中介的好处

策略4：进度提示
  - H5表单："第1步/共4步"
  - 降低心理负担
  - 避免中途放弃
```

### 2.4 数据校验规则

**2.4.1 前端校验（实时反馈）**
```javascript
// 营收与开票逻辑一致性
if (营收 < 开票) {
  警告："开票金额通常不大于营收，请核实"
}

// 票税比合理性
const 票税比 = 年开票 / 年纳税;
if (票税比 < 5 || 票税比 > 50) {
  警告："票税比异常（正常5-20倍），请核实"
}

// 成立时间合理性
if (成立时间 > 今天) {
  错误："成立时间不能晚于今天"
}

// 房产估值合理性
if (房产估值 < 50万 || 房产估值 > 5000万) {
  警告："房产估值似乎异常，请核实"
}
```

**2.4.2 后端校验（严格验证）**
```python
# 必填字段检查
required_fields = ['企业名称', '所在城市', '所属行业', '成立时间']
for field in required_fields:
    if not data.get(field):
        raise ValidationError(f"{field}不能为空")

# 数据类型检查
if not isinstance(年营业收入, (int, float)):
    raise ValidationError("年营业收入必须是数字")

# 枚举值检查
if 所属行业 not in VALID_INDUSTRIES:
    raise ValidationError("所属行业不在有效列表中")

# 逻辑一致性检查（严格）
if 年开票 > 年营业收入 * 1.2:
    raise ValidationError("开票金额不应超过营收的120%")
```

## 三、产品库（20个银行产品）

### 3.1 数据来源渠道

**渠道1：银行官网**（40%，可信度⭐⭐⭐⭐⭐）
```yaml
优势：
  ✅ 权威性最高（官方发布）
  ✅ 免费获取
  ✅ 可随时查询

劣势：
  ⚠️ 更新滞后（政策调整可能未及时更新官网）
  ⚠️ 信息不全（部分细节未公开）
  ⚠️ 需要人工整理

获取方式：
  1. 定期访问银行官网（每月1次）
  2. 查看"企业贷款"/"小微贷款"栏目
  3. 下载产品手册/说明书
  4. 整理为标准化格式

工作量：
  - 初期：20个产品 × 1小时 = 20小时
  - 维护：20个产品 × 30分钟/月 = 10小时/月
```

**渠道2：客户经理**（30%，可信度⭐⭐⭐⭐）
```yaml
优势：
  ✅ 信息最新（实际审批标准）
  ✅ 细节完整（包括隐性要求）
  ✅ 可咨询疑问

劣势：
  ⚠️ 需要建立关系
  ⚠️ 信息可能口径不一致
  ⚠️ 部分客户经理不愿透露

获取方式：
  1. 联系银行客户经理（每季度1次）
  2. 询问产品最新政策
  3. 确认审批标准、利率范围
  4. 记录变化并更新产品库

工作量：
  - 初期：建立关系（5-10个客户经理）
  - 维护：每季度1次沟通（2小时/季度）
```

```yaml
优势：
  ✅ 实时发现政策变化
  ✅ 一线最早感知

劣势：

获取方式：
```

### 3.3 产品库维护演进

**V0阶段（0-6月）：人工手动维护**
```yaml
维护方式：
  - 每月检查1次官网（20个产品 × 30分钟）
  - 每季度联系1次客户经理（重点产品）
  - 中介反馈即时记录

成本：
  - 10-15小时/月（1人兼职）

工作流程：
  1. 每月1日：访问20个银行官网
  2. 对比上月数据：发现差异
  3. 标记变化：需要核实的产品
  4. 联系客户经理：确认变化
  5. 更新产品库：写入新数据
  6. 通知中介：政策变化提醒

验收标准：
  - 20个产品信息完整
  - 每月更新1次
  - 核验日期清晰
```

**V1阶段（6-12月）：众包维护机制**
```yaml
维护方式：
  - 中介每次使用产品，系统提示：
    "这个产品信息是3个月前的，有变化吗？"
  - 中介更新 → 系统给积分/返佣优惠
  - 人工复核 → 确认后更新产品库

成本：
  - 5小时/月（仅复核）

优势：
  - 中介是一线，最早发现政策变化
  - 积分激励成本低
  - 数据更新频率高（从每月1次→每周多次）

激励机制：
  - 更新1条产品信息 = 20积分
  - 积分兑换：VIP功能、返佣优惠
  - 更新被采纳 → 额外奖励
```

**V2阶段（12月+）：爬虫自动化**
```yaml
维护方式：
  - 定时爬取银行官网（每周1次）
  - 对比差异 → 自动标记"需人工核实"
  - 人工只需复核差异部分

成本：
  - 2小时/月（仅复核差异）

技术实现：
  - Python爬虫（Scrapy）
  - 文本对比（difflib）
  - 变化提醒（钉钉/企微）
  - 自动记录核验日期

工作流程：
  1. 爬虫每周六运行
  2. 爬取20个银行官网产品页
  3. 对比上周数据
  4. 发现差异 → 钉钉通知
  5. 人工复核 → 确认变化
  6. 更新产品库 → 通知中介
```

## 五、结果库（数据飞轮核心）⭐⭐⭐⭐⭐

### 5.1 为什么是生死线

**数据飞轮原理**：
```
初始状态（V0）：
  系统预测：产品P001适合客户，成功率85%
  依据：银行官网政策 + 诊断算法（静态规则）

    ↓

数据飞轮启动（V1，回填率>50%）：
  1. 系统推荐产品P001 → 100次
  2. 中介回填结果 → 80次通过，20次拒绝
  3. 分析拒绝原因 → 发现规律：
     - 流水波动>30%的客户，通过率只有60%
     - 流水波动<30%的客户，通过率95%
  4. 系统调整规则：
     - 流水波动>30% → 降低推荐优先级
     - 流水波动<30% → 提高推荐优先级
  5. 结果：准确率从85% → 92%

    ↓

持续优化（V2，回填率>80%）：
  1. 积累1000条结果
  2. 发现更多隐性规则：
     - 查询次数>8次 + 流水波动>30% → 通过率只有40%
     - 查询次数<6次 + 流水波动<20% → 通过率98%
  3. 系统继续优化规则
  4. 准确率持续提升：92% → 95%
```

**竞品对比**：
```
竞品A（无结果库）：
  - 3年前：匹配准确率70%
  - 3年后：匹配准确率还是70%
  - 原因：没有数据飞轮，系统不会变聪明

我们（有结果库）：
  - V0：匹配准确率85%
  - V1（6个月）：匹配准确率92%（+7%）
  - V2（2年）：匹配准确率95%（+10%）
  - 原因：数据飞轮持续优化

结论：
  - 结果库是核心竞争力
  - 回填率<50% = 数据飞轮无法启动
  - 回填率>80% = 远超竞品
```

```javascript
if (拒绝 && !拒绝原因) {
  错误："拒绝时必须选择原因（六宫格归因）"
}
```

**第二关：后端容错（宽容处理）**
```python
# 客户档案库
if 纳税信用等级 == "不清楚":
    纳税信用等级 = "C"  # 按保守估计

# 产品库
if 信息来源 == "第三方平台":
    可信度 = "中"  # 降低可信度

# 方案库
if 样本量 < 30:
    成功率 = 成功率 - 10%  # 保守估计

# 结果库
if 实际额度 is None:
    实际额度 = 预测额度  # 默认值
```
